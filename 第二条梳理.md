可能被问到的问题：
- **这条链路的整体延迟在哪里？** 客户端视角：HTTP 请求延迟 = 缓存读取 + Lua 脚本 + MQ 投递。MQ 投递是异步的（mqSendExecutor），所以 HTTP 响应不等落库。用户拿到 messageId 和 seq 就可以在 UI 上先展示"发送中"
- **如果 MQ 投递失败，客户端已经拿到了 messageId，但消息没落库怎么办？** 答：onException 回调写 im_msg_failover 本地消息表，定时任务 MsgFailoverRetryTask 每 10 秒扫描重投。客户端通过 /sync 接口拉取时，如果 seq 对应消息不存在，返回 tombstone 标记
- **为什么 HTTP 响应先返回再异步落库，而不是同步落库后返回？** 答：同步落库会把 MySQL INSERT 延迟加到请求链路上。压测发现同步落库时 P99 高达 48s（第一轮瓶颈），异步后降到 80ms

二、幂等去重方案深挖

2.1 Redis Lua 原子去重 + seq 分配（MessageServiceImpl.java:562-572）

```lua
if redis.call('set', KEYS[1], '1', 'PX', ARGV[1], 'NX') then
  return redis.call('incr', KEYS[2])
end
return -1
```

参数说明：
| 参数 | 值 | 含义 |
|------|-----|------|
| KEYS[1] | `im:msg:dedup:{sessionId}:{clientMsgId}` | 去重键 |
| KEYS[2] | `im:seq:{sessionId}` | 会话级自增序号 |
| ARGV[1] | `300000`（5分钟，毫秒） | 去重键 TTL |
| 返回值 > 0 | 新 seq | 首次发送，分配 seq |
| 返回值 = -1 | — | 重复消息，拒绝 |

可能被问到的问题：

- **为什么去重和 seq 分配要放在同一个 Lua 脚本里？** （答：原子性。如果分成两步：先 SETNX 去重，再 INCR seq，两步之间如果进程崩了——去重 key 已设但 seq 未分配——会导致 seq 空洞。合在一个 Lua 里，要么两步都执行，要么都不执行）
- **SET NX PX 是什么语义？** （答：NX = Not eXist，只有 key 不存在时才设置成功。PX = 以毫秒设置过期时间。等价于 `SETNX + PEXPIRE` 的原子版本）
- **去重 TTL 5 分钟怎么定的？** （答：客户端重试窗口。如果网络抖动导致客户端 5 分钟内重发同一条消息，服务端能识别为重复。5 分钟后去重 key 过期，如果客户端还在重试，会被当作新消息——但此时 clientMsgId 应该已经不同了。这个 TTL 要大于客户端的最大重试间隔）
- **INCR 的 key `im:seq:{sessionId}` 没有设置过期时间，会不会一直占 Redis 内存？** （答：设计如此。seq 是会话级的单调递增计数器，必须持久化。只要会话存在，seq 就需要在。可以定期清理已删除会话的 key）
- **为什么用会话级 seq 而不是全局 seq？** （答：会话级 seq 的优势：1) 不同会话的 seq 互不影响，减少竞争；2) 客户端按 sessionId 维度做增量同步，会话级 seq 天然对齐；3) 全局 seq 在高并发下是单点热 key）
- **seq 是严格递增的吗？会不会出现空洞？** （答：INCR 保证严格递增。但空洞可能出现：Lua 返回了 seq=100 但后续 MQ 投递失败，消息没落库，seq=100 就是空洞。客户端 /sync 拉取时遇到空洞返回 tombstone，告诉客户端"这个 seq 不存在，不用等了"）
- **如果 Redis 宕机，seq 全丢了怎么办？** （答：INCR 只在内存中，需要 Redis 持久化（AOF/RDB）。如果丢了，恢复方式：从 MySQL im_message 表查 MAX(seq) WHERE session_id=xxx，回写 Redis。但这段时间内新消息可能 seq 冲突）

三、会话级有序投递

3.1 RocketMQ 顺序消息原理

你代码中两处使用了顺序发送：
1. `asyncSendOrderly("IM_MSG_STORE", mqMessage, sessionId)` — MessageServiceImpl.java:153
2. `syncSendOrderly("IM_CHAT:" + mqTag, mqMessage, sessionId)` — MessageStoreListener.java:88

可能被问到的问题：

- **asyncSendOrderly 的 hashKey 参数 sessionId 是干什么的？** （答：RocketMQ 根据 hashKey 做 hash 取模选择 MessageQueue。相同 sessionId 的消息会被路由到同一个 MessageQueue，保证在同一个 Queue 内 FIFO 顺序消费）
- **顺序消费 ORDERLY vs 并发消费 CONCURRENTLY 的区别？** （答：ORDERLY = 同一个 Queue 内的消息由同一个消费线程串行处理，保证消费顺序。CONCURRENTLY = 同一个 Queue 的消息可以被多个线程并发消费，不保证顺序）
- **你的 IM_MSG_STORE 消费者用的是 CONCURRENTLY（MessageStoreListener.java:35），为什么发送端用了 Orderly 但消费端不用 Orderly？** （答：关键洞察！发送端用 sendOrderly 保证同一 session 的消息落到同一 Queue，但消费端用 CONCURRENTLY 是因为：落库操作彼此独立，消息已经有 seq 标识顺序，不需要消费端严格串行。并发消费吞吐量更高，落库后推送到 Netty 节点时也是按 sessionId 做 sendOrderly，最终用户端按 seq 排序即可）
- **如果不用 sendOrderly 会怎样？** （答：同一 session 的消息可能被分散到不同 Queue，多个消费线程并发处理时，可能出现 seq=2 的消息比 seq=1 先落库。虽然最终客户端按 seq 排序不影响展示，但中间状态下 /sync 接口可能返回不连续的 seq 列表）
- **asyncSendOrderly 和 syncSendOrderly 的区别？** 
  - `asyncSendOrderly`：异步发送，立即返回，通过 SendCallback 回调通知结果。用于发送入口 → IM_MSG_STORE，不阻塞 HTTP 请求线程
  - `syncSendOrderly`：同步发送，阻塞等待 Broker 确认。用于 MessageStoreListener → IM_CHAT，确保推送一定进入 MQ

3.2 MQ Topic 设计

| Topic | 用途 | 生产者 | 消费者 | 消费模式 |
|-------|------|--------|--------|---------|
| IM_MSG_STORE | 消息落库 | MessageServiceImpl | MessageStoreListener | CONCURRENTLY, 20线程 |
| IM_CHAT:{nodeId} | 定向推送到 Netty 节点 | MessageStoreListener | NettyPushMessageListener (RTC) | CONCURRENTLY |
| IM_CHAT:OFFLINE | 离线消息回流 | MessageStoreListener | OfflineService | CONCURRENTLY |
| IM_NOTIFY:{nodeId} | 通知推送 | NettyMessageService | NotifyPushMessageListener (RTC) | CONCURRENTLY |
| AI_AGENT_REQUEST:{tag} | AI 助手触发 | MessageServiceImpl | AI Agent Service | — |

可能被问到的问题：

- **为什么 IM_MSG_STORE 和 IM_CHAT 分两个 Topic，而不是一个 Topic 内先落库再推送？** （答：解耦。落库是 Messaging 服务的职责，推送是 RTC 服务的职责。分 Topic 后两个服务独立消费，互不阻塞。如果合一个 Topic，落库慢会拖慢推送）
- **IM_CHAT:{nodeId} 这个 Tag 式路由的含义？** （答：每个 Netty 节点只消费 tag=自己nodeId 的消息。通过 selectorExpression 过滤，避免节点收到不属于自己管理的用户的消息）
- **IM_MSG_STORE 的 consumeThreadNumber=20 怎么定的？** （答：经验值。MySQL INSERT 是 I/O 密集型操作，20 个消费线程可以充分利用数据库连接池（HikariCP max=30）。如果消费线程 > 连接池，线程会等待连接，反而降低吞吐）

3.3 消息投递语义

- **你的消息链路是"至少一次投递"还是"恰好一次"？** （答：至少一次投递 + 消费端去重 = 效果上接近恰好一次。具体保障机制：
  1. Redis Lua 去重：发送端层面防止 clientMsgId 重复
  2. MQ 至少一次投递：RocketMQ 消费失败会重试（max-reconsume-times=16 默认值）
  3. 数据库主键去重：messageId 唯一约束，INSERT 重复忽略
  4. 客户端 seq 去重：客户端按 seq 排序展示，重复 seq 忽略）
- **RocketMQ 的消息重试机制是怎样的？** （答：消费返回 RECONSUME_LATER 或抛异常 → Broker 重新投递。重试间隔递增：10s, 30s, 1min, 2min...最多重试 16 次后进死信队列 %DLQ%）

可能被问到的问题：

- **为什么先落库再推送，而不是反过来？** （答：保证消息持久化优先。如果先推送后落库，推送成功但落库失败，消息会丢失——用户 A 看到了消息但重连后 /sync 拉不到）
- **storeMessage 中 entity.setContent(JSON.toJSONString(appMessage.getBody()))，body 是 Object 类型序列化为 String 存入 MySQL。这有什么问题？** （答：body 可能是各种类型（文本、图片、文件的 JSON 结构）。统一序列化为 String 存 TEXT/LONGTEXT 字段，查询时需要反序列化。如果内容很大（比如转发消息），MySQL TEXT 有 64KB 限制）
- **entity.setCreateTime(new Date()) 而不是用 appMessage.getCreatedAt()？** （答：代码中两个时间都设了。appMessage.getCreatedAt() 是发送时间的格式化字符串（用于前端展示），entity.setCreateTime() 是数据库记录时间。两者可能有几毫秒到几秒的差异（MQ 传输延迟），以数据库时间为准）

5.1 MsgFailover 方案设计

当 MQ 投递失败时，消息不丢：

5.2 MsgFailoverRetryTask（MsgFailoverRetryTask.java:38-141）

```
@Scheduled(fixedDelay = 10_000L)  — 每 10 秒执行一次
1. Redisson 分布式锁 im:task:msg-failover-retry（tryLock 非阻塞）
2. 查询 status=0 的记录，LIMIT 500
3. 逐条处理：
   a. retryCount >= 5 → 标记为死信 (status=-1)
   b. 解析 payload → GatewayPushPacket
   c. multiGet 查询目标用户路由
   d. 每个用户 syncSendOrderly("IM_CHAT:{nodeId}", ...)
   e. 全部成功 → status=1; 部分失败 → retryCount++
4. 批次处理超过 8 秒则 break，防止长时间占锁
```

可能被问到的问题：

- **为什么用 Redisson 分布式锁？** （答：如果 Messaging 部署多实例，@Scheduled 每个实例都会触发。分布式锁保证同一时刻只有一个实例在执行重试任务，避免重复投递）
- **tryLock() 不传参数是什么语义？** （答：waitTime=0, leaseTime=Watchdog默认30s。不等待，拿不到锁立即返回 false。leaseTime 由 Watchdog 自动续期，直到方法结束调用 unlock()）
- **MAX_RETRY_COUNT=5，超过后标记为死信。死信消息怎么处理？** （答：当前代码只是 status=-1 存在数据库。生产环境应该有告警 + 人工介入。也可以考虑发到 DLQ Topic 由专门的死信消费者处理）
- **BATCH_DEADLINE_MS=8000，为什么不处理完所有 500 条？** （答：防止单批处理时间过长。如果某些 MQ 投递超时 3s/条，500 条可能需要 25 分钟。8 秒截断后下一轮继续，保证定时任务不积压）
- **failover 只在 MQ 投递失败时写入。如果 MessageServiceImpl 到 MQ 之间的整个 mqSendExecutor 线程池满了呢？** （答：CallerRunsPolicy 降级——由调用方线程（Tomcat 线程）直接执行 asyncSendOrderly。如果 Tomcat 线程也超时，会抛异常走 catch 块，redisTemplate.delete(dedupKey) 清除去重 key，下次重试可以重新分配 seq）

5.3 本地消息表 vs 事务消息

面试官可能追问：**为什么不用 RocketMQ 事务消息保证投递？**

（答：事务消息适合"本地事务 + MQ 投递"需要原子性的场景。你的 sendMessage 没有本地事务——seq 是 Redis 分配的，落库是异步的。发送端只需要"尽力投递 MQ + 失败兜底"，本地消息表更轻量。红包场景才真正需要事务消息，因为有"余额扣减 + MQ 投递"的原子性需求）

六、多级缓存架构

6.1 Cache-Aside + 防雪崩策略

```java
// MessageServiceImpl.java:221-235
private User getUserWithCache(Long userId) {
    String key = CACHE_USER_PREFIX + userId;
    String json = redisTemplate.opsForValue().get(key);
    if (StringUtils.isNotBlank(json)) {
        return JSON.parseObject(json, User.class);       // 命中缓存
    }
    User user = userService.getById(userId);              // 穿透到 MySQL
    if (user != null) {
        long expire = 24L + ThreadLocalRandom.current().nextInt(24);  // 24~48小时随机
        redisTemplate.opsForValue().set(key, JSON.toJSONString(user), expire, TimeUnit.HOURS);
    }
    return user;
}
```

缓存 Key 设计：
| Key 模式 | 存储内容 | TTL |
|----------|---------|-----|
| `im:cache:user:{userId}` | User JSON | 24~48h 随机 |
| `im:cache:friend:{userId}:{friendId}` | Friend JSON | 24~48h 随机 |
| `im:cache:session:{sessionId}` | Session JSON | 24~48h 随机 |
| `im:cache:session:members:{sessionId}` | List\<Long\> JSON | 24~48h 随机 |

可能被问到的问题：

- **为什么 TTL 用 24~48 小时随机而不是固定值？** （答：防止缓存雪崩。如果 1000 个用户的缓存都是 24 小时后同时过期，那一瞬间所有请求穿透到 MySQL。随机 TTL 把过期时间打散）
- **Cache-Aside 模式有什么一致性问题？** （答：经典问题——先更新 DB 还是先删缓存？你的代码是读时写入（懒加载），不在写时更新缓存。问题场景：用户改了昵称 → MySQL 更新了 → 但缓存里还是旧昵称 → 最长 48 小时才过期。如果要求实时性，需要在更新 User 时主动删缓存）
- **getSingleChatCacheSnapshot 为什么用 multiGet 一次取 3 个 key？** （答：减少 Redis RTT。单聊发消息需要查发送者、接收者、好友关系三个维度，multiGet 一次网络往返取回所有数据，比 3 次 GET 快 2 个 RTT）
- **缓存穿透怎么防？** （答：当前代码没有处理——如果 userId 不存在，每次都穿透到 MySQL。可以缓存空值（设短 TTL）或用布隆过滤器）
- **缓存击穿怎么防？** （答：当前没有处理——热 key 过期瞬间大量请求穿透。可以用互斥锁（Redisson tryLock）或逻辑过期（永不过期 + 后台异步刷新））

七、消息同步与 ACK 机制

7.1 消息同步接口（MessageServiceImpl.java:574-607）

```
GET /api/message/sync?sessionId=xxx&beginSeq=10&endSeq=60
  ↓
1. actualEndSeq = min(endSeq, beginSeq + 500)  — 单次最多拉 500 条
2. selectBySeqRange(sessionId, beginSeq, actualEndSeq)
3. 遍历 [beginSeq+1, actualEndSeq]：
   - 有消息 → SyncMessageVo(isTombstone=false)
   - 无消息 → SyncMessageVo(isTombstone=true, status=-1)
```

可能被问到的问题：

- **tombstone 消息是什么？为什么需要它？** （答：seq 空洞标记。当客户端拉取 seq=10~20 的消息时，如果 seq=15 的消息因为 MQ 投递失败没有落库，不返回 tombstone 的话，客户端会一直等 seq=15。返回 tombstone 告诉客户端"这个 seq 没有消息，跳过它"）
- **单次最多拉 500 条的限制怎么定的？** （答：防止一次返回过多数据导致响应体过大、MySQL 查询慢。客户端需要分页拉取：第一次 0~500，第二次 500~1000...）
- **selectBySeqRange 的 SQL 是 WHERE seq > beginSeq AND seq <= endSeq AND status = 0 ORDER BY seq ASC，有没有合适的索引？** （答：需要联合索引 (session_id, seq)，或者 (session_id, status, seq)。没有索引的话全表扫描会很慢）

7.2 ACK 机制

两种 ACK 方式：
1. **按 seq ACK**：`POST /ack?sessionId=xxx&seq=100` → updateLastAckSeq(userId, sessionId, 100)
2. **按 messageId ACK**：`POST /ack?messageId=msg_xxx` → 回查消息获取 sessionId+seq → updateLastAckSeq

updateLastAckSeq 的关键 SQL（UserSessionServiceImpl.java:30-36）：
```java
wrapper.eq(UserSession::getUserId, userId)
    .eq(UserSession::getSessionId, sessionId)
    .lt(UserSession::getLastAckSeq, seq)   // 只增不减！
    .set(UserSession::getLastAckSeq, seq);
```

可能被问到的问题：

- **`.lt(lastAckSeq, seq)` 是什么意思？** （答：只有当新 seq > 当前 lastAckSeq 时才更新。防止乱序 ACK 导致 lastAckSeq 回退。比如客户端先收到 seq=10 再收到 seq=8 并分别 ACK，如果不加 lt 条件，后到的 ACK(seq=8) 会把 lastAckSeq 从 10 退回 8）
- **ACK 是同步 HTTP 调用，在 RTC 的 Business 线程里走 Feign 调用。简历里提到的"伪同步 ACK"瓶颈是什么？** （答：旧版本中，RTC 的 MessageInboundHandler 收到客户端 ACK 消息后，通过 Feign 同步调用 Messaging 的 /ack 接口。如果 Messaging 响应慢（比如 MySQL 更新慢），Business 线程会被阻塞，导致其他消息的处理也堆积）
- **登录/重连时的离线消息补齐流程？**
  ```
  1. 客户端重连成功
  2. 调用 GET /sessions/ackSeqMap → {session1: lastAckSeq=100, session2: lastAckSeq=50}
  3. 对每个 session 调用 GET /maxSeq → session1 当前 seq=120
  4. 对每个 session 调用 GET /sync?beginSeq=100&endSeq=120
  5. 拉取到的消息渲染 UI
  6. 发 ACK 更新 lastAckSeq
  ```

八、消息 ID 生成方案（IdGenerator.java）

8.1 雪花算法实现

```
64位 = 1(符号位) + 41(时间戳) + 5(datacenter) + 5(worker) + 12(序列号)
```

| 组件 | 位数 | 范围 | 你的实现 |
|------|------|------|---------|
| 时间戳 | 41 bit | 约 69 年 | 自定义纪元 2023-01-01 |
| datacenterId | 5 bit | 0~31 | hostname.hashCode() % 32 |
| workerId | 5 bit | 0~31 | IP.hashCode() % 32 |
| 序列号 | 12 bit | 0~4095 | 毫秒内自增 |

最终 messageId = `"msg_" + snowflakeId`

可能被问到的问题：

- **时钟回拨怎么处理？** （答：你实现了两级防御：1) 5ms 内的轻微回拨 → sleep(offset*2) 等待追上；2) 超过 5ms 的严重回拨 → 直接抛异常拒绝生成。面试亮点：可以提到第三种策略——改变 workerId 继续生成，牺牲全局有序性保可用性）
- **nextId() 是 synchronized 的，高并发下会不会成为瓶颈？** （答：单线程每毫秒可以生成 4096 个 ID，即 400 万 QPS。对于 1000 RPS 的 IM 场景远远够用。如果真成瓶颈，可以用 ThreadLocal 版或预分配号段）
- **workerId 和 datacenterId 基于 IP 和 hostname 的 hashCode，有没有碰撞风险？** （答：有。hash 值 mod 32 的碰撞概率不低。生产环境应该从注册中心（Nacos/ZooKeeper）动态分配 workerId，确保全局唯一）
- **为什么不直接用 UUID？** （答：UUID 无序，作为 MySQL 主键会导致 B+ 树频繁分裂，INSERT 性能差。雪花 ID 趋势递增，适合做主键。另外雪花 ID 内嵌时间信息，可以反解出生成时间）

九、MQ 发送专用线程池

9.1 MqSendExecutorConfig（MqSendExecutorConfig.java:19-29）

```java
new ThreadPoolExecutor(
    16,                              // 核心线程数
    64,                              // 最大线程数
    60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(8192),  // 有界队列
    r -> new Thread(r, "mq-send-" + r.hashCode()),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

可能被问到的问题：

- **为什么要单独抽一个线程池？直接在 Tomcat 线程中 asyncSend 不行吗？** （答：asyncSendOrderly 底层在 RocketMQ producer 内部有一个等待队列。当 Broker 繁忙或网络抖动时，producer 队列满了会阻塞调用线程。如果调用线程是 Tomcat 请求线程，就会导致 HTTP 接口超时。用独立线程池隔离，Tomcat 线程 execute() 后立即返回）
- **这正是简历中提到的"伪同步 ACK"瓶颈的根本原因吗？** （答：不完全是。伪同步 ACK 是指 RTC 端 Feign 同步调用 Messaging /ack 接口阻塞了 Business 线程。mqSendExecutor 解决的是 Messaging 端 MQ 投递阻塞 Tomcat 线程的问题。两个是不同的瓶颈点，都在 3 轮优化中被定位和解决）
- **CallerRunsPolicy 在什么场景触发？** （答：当核心线程 16 个都在忙 + 队列 8192 已满 + 非核心线程扩张到 64 个也都在忙时，新任务会由提交任务的线程（Tomcat 线程）直接执行。这是一种优雅降级——宁可让 Tomcat 线程慢一点，也不丢任务。对于用户来说，这次请求的响应时间会变长，但消息不丢）
- **核心 16、最大 64、队列 8192 这些参数怎么定的？** （答：核心 16 ≈ MQ 发送的并发度，每条消息 asyncSendOrderly 约 1~3ms，16 线程可以支撑 5000~16000 TPS。队列 8192 是缓冲区，应对突发流量。最大 64 是极端情况下的弹性。这些参数应该根据压测数据调整）

可能被问到的问题：

- **max=400 是怎么定的？** （答：2C8G 机器上，每个请求主要是 Redis 读 + MQ 异步投递，CPU 消耗不高但有网络等待。400 线程可以支撑较高并发。如果 CPU 密集型应该 = 2*CPU 核数）

- **压测 1000 RPS 时 Tomcat 线程够用吗？** （答：如果每个请求处理时间 < 400ms，400 线程可以支撑 400/0.4=1000 RPS。实际请求处理时间（Redis 读 + 异步 MQ）约 2~5ms，所以 400 线程远远够用。瓶颈不在 Tomcat 线程数）

11.2 三轮优化详情

**第一轮：491 RPS / P99 48s**

问题：@Transactional 事务长占 MySQL 连接

定位过程：
1. wrk2 输出 actual RPS 远低于 target 1000，P99 延迟 48 秒

**第二轮：~700 RPS，P99 仍偏高**

问题：asyncSendOrderly 在 producer 队列满时阻塞 Tomcat 线程

1. P99 降了但仍不稳定，偶尔飙高
2. jstack → 部分 Tomcat 线程阻塞在 RocketMQ DefaultMQProducerImpl.send() 内部的锁等待
3. 根因：asyncSendOrderly 底层获取 Queue 的锁时，如果 Broker 响应慢，会阻塞调用线程

修复：引入 mqSendExecutor 专用线程池，Tomcat 线程 execute() 后立即返回

**第三轮：~900 RPS → 994 RPS / P99 80ms**

定位过程：
1. RPS 接近 1000 但偶尔掉到 900 以下
2. 监控 Redis 连接数 → Lettuce 连接池耗尽，线程等待连接
3. 根因：Spring Boot 默认 max-active=8，高并发下不够用

十三、RocketMQ 八股（高频考点）

13.1 RocketMQ 基础架构
- **NameServer**：无状态路由注册中心，Broker 每 30s 上报心跳
- **Broker**：消息存储和转发，分 Master/Slave
- **Producer**：消息生产者，从 NameServer 获取路由信息
- **Consumer**：消息消费者，支持 Push 和 Pull 两种模式

13.2 你的 RocketMQ 配置

```yaml
rocketmq:
  name-server: 172.30.233.168:9876
  producer:
    group: im-message-producer-group
    send-message-timeout: 1000    # 发送超时 1 秒
```

可能被问到的问题：

- **send-message-timeout=1000ms，如果 Broker 1 秒没响应怎么办？** （答：抛 RemotingTooMuchRequestException。在 asyncSendOrderly 场景下会回调 onException → 写 failover 表）
- **Producer Group 的作用？** （答：事务消息回查时 Broker 需要找到同组的 Producer 来执行 checkLocalTransaction。普通消息场景下 Group 主要用于标识和管理）
- **RocketMQ 消息的存储结构？** （答：CommitLog + ConsumeQueue + IndexFile。CommitLog 是顺序写的消息日志文件，ConsumeQueue 是按 Topic/Queue 维度的索引，IndexFile 是按 Key 的哈希索引）
- **如何保证消息不丢？** （答：三层保障：1) Producer 同步发送确认；2) Broker 刷盘策略（同步刷盘 vs 异步刷盘）；3) Consumer ACK 确认。你的代码中发送端用 asyncSendOrderly + failover 兜底，消费端异常时抛异常触发重试）

13.3 顺序消息的实现原理

```
Producer 端：hashKey(sessionId) % queueSize → 选择固定 MessageQueue
Broker 端：同一 Queue 内消息严格 FIFO
Consumer 端（ORDERLY）：加锁消费，同一 Queue 只能被一个消费线程处理
Consumer 端（CONCURRENTLY）：不加锁，同一 Queue 可被多线程并发处理
```

- **你用 CONCURRENTLY 消费 IM_MSG_STORE，那消息落库顺序不是严格按 seq 的？** （答：是的，不严格有序。但这没关系——seq 已经在发送端由 Redis 分配好了，落库时 seq 字段值是确定的，只是 INSERT 的先后顺序不保证。客户端 /sync 按 seq ASC 查询，最终展示是有序的）

可能被问到的问题：

- **为什么 message_id 用 VARCHAR 而不是 BIGINT？** （答：因为 messageId = "msg_" + snowflakeId，带了字符串前缀。如果纯数字可以用 BIGINT 更高效。VARCHAR 主键在 B+ 树中比较开销更大）
- **content 字段用 TEXT 类型，有什么限制？** （答：MySQL TEXT 最大 64KB。如果消息内容含图片 URL + 缩略图 base64 + 富文本，可能超限。LONGTEXT 最大 4GB 但会影响性能。IM 场景通常限制消息体大小）
- **im_message 表数据量增长很快，怎么做分表？** （答：按 session_id 或 create_time 做水平分表。按 session_id 分表后，同一会话的消息在同一张表，/sync 查询不需要跨表。可以用 ShardingSphere 实现）

在 SendMsgRequest 基础上增加：

15.4 推送包装 GatewayPushPacket

```java
public class GatewayPushPacket {
    private List<Long> targetUserIds;  // Netty 用来查 Channel

}
```

十六、选型对比（面试高频题）

16.1 消息存储：写扩散 vs 读扩散

| 维度 | 写扩散（你的方案） | 读扩散 |
|------|-------------------|--------|
| 发送 | 每个接收者写一条离线记录 | 只写一条原始消息 |
| 拉取 | 直接查自己的消息列表 | 查消息表 + 过滤 |
| 存储 | 冗余大（群聊 N 人写 N 条） | 存储小 |
| 适用 | 小群（<500人）、IM 主流方案 | 大群（朋友圈、微博） |

你的方案：IM_MSG_STORE 落库一条原始消息 + 按路由推送到各节点/离线。属于 **读扩散+推送** 的混合模型——原始消息只存一条（读扩散），但推送时按节点分发（推扩散）。

16.2 MQ 选型：RocketMQ vs Kafka vs RabbitMQ

| 维度 | RocketMQ（你的选择） | Kafka | RabbitMQ |
|------|---------------------|-------|----------|
| 延迟 | 毫秒级 | 毫秒级（batch后）| 微秒级 |
| 顺序消息 | 原生支持 | Partition 级别 | 不直接支持 |
| 事务消息 | 原生支持 | 不支持 | 不支持 |
| 延迟消息 | 原生支持（18级） | 不支持 | 插件支持 |
| 消费模式 | Push + Pull | Pull | Push |
| 适用 | IM/金融/电商 | 日志/大数据 | 轻量业务 |

为什么选 RocketMQ？（答：1) 原生顺序消息——IM 需要会话级有序；2) 原生事务消息——红包链路需要；3) 延迟消息——红包过期需要；4) 阿里系技术栈生态好）

16.3 序号生成：Redis INCR vs 数据库自增 vs 号段模式

| 方案 | 性能 | 一致性 | 容灾 |
|------|------|--------|------|
| Redis INCR（你的方案） | 极高（10万+ QPS） | 依赖 Redis 持久化 | Redis 宕机丢数据 |
| 数据库自增 | 中 | 强一致 | 依赖主从切换 |
| 号段模式（Leaf） | 高 | 段内强一致 | 段用完需续号 |

十八、生产环境优化与架构演进

面试官可能在以上基础上继续追问：

- **消息存储量增长怎么办？** （答：im_message 按月分表或按 session_id hash 分表。冷数据归档到 HBase/TiKV。热数据保留最近 3 个月）
- **缓存和数据库的一致性怎么保证？** （答：当前方案是读时写入 + 随机 TTL。如果需要实时性，可以引入 Canal 监听 MySQL binlog → 更新/删除 Redis 缓存）
- **如果 Redis 宕机，整个消息链路会怎样？** （答：1) 去重失败 → 所有消息被视为重复返回 clientMsgId；2) seq 分配失败 → sendMessage 抛异常；3) 路由查询失败 → 所有推送走 OFFLINE。Redis 是单点强依赖，需要 Redis Sentinel 或 Cluster）
- **如何支持消息撤回？** （答：UPDATE im_message SET status=1 WHERE message_id=xxx。然后推送一条"撤回通知"给所有会话成员。客户端收到后从 UI 删除该消息。撤回有时间限制（通常 2 分钟））
- **如何支持已读回执？** （答：接收者 ACK 后，服务端推送 ACK 通知给发送者。单聊简单；群聊需要统计"N 人已读"，可以用 Redis HyperLogLog 或 Bitmap 计数）
- **如何支持万人群？** （答：当前方案一条消息要 multiGet 所有群成员路由 + 按节点分组推送。万人群意味着 multiGet 10000 个 key、可能分发到几十个 Netty 节点。优化：1) 群成员路由缓存到本地 + 增量更新；2) 推送时用广播而非单播；3) 按群维度做 Topic 分区）
