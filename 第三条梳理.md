一、离线消息全链路总览（面试官第一问：画出你的离线消息链路）

```
消息发送链路 (Messaging Service)
  ↓
MessageStoreListener 落库后路由推送
  ├── 查 Redis im:route:{userId}
  ├── 在线 → syncSendOrderly("IM_CHAT:{nodeId}", packet, sessionId)
  └── 离线 → syncSendOrderly("IM_CHAT:OFFLINE", packet, sessionId)
         ↓
OfflineService: OfflineMessageListener [topic=IM_CHAT, tag=OFFLINE]
  ├── 1. 解析 GatewayPushPacket → AppMessage
  ├── 2. 遍历 targetUserIds:
  │    ├── storeOfflineMessage(receiverId, wsPayload)
  │    │    ├── storeToRedis()  → ZSet im:offline:{userId}:{sessionId}, score=seq, TTL=3天
  │    │    └── storeToMySQL()  → INSERT offline_message, expire_at=now+7天
  │    └── incrementUnreadCount(receiverId, sessionId, messageId)
  │         ├── Redis HINCRBY im:unread:{userId} sessionId 1
  │         └── MySQL INSERT ON DUPLICATE KEY UPDATE unread_count+1
  └── 3. 消费成功 / 失败重试

客户端重连补拉:
  1. GET /sessions/ackSeqMap → {session1: lastAckSeq=100, session2: lastAckSeq=50}
  2. POST /api/offline/pull → PullOfflineMessageRequest{sessionId, lastSeq, pageSize}
  3. 热数据优先：Redis ZSet rangeByScore(lastSeq, +∞)
  4. 冷数据兜底：MySQL SELECT WHERE seq > lastSeq ORDER BY seq ASC LIMIT pageSize
  5. 返回 PullOfflineMessageResponse{messages, hasMore, total, currentTime}
```

可能被问到的问题：

- **为什么离线消息要单独做一个 OfflineService，而不是放在 Messaging 里？** （答：职责分离。Messaging 负责消息的发送、seq 分配、落主消息表。OfflineService 负责离线存储、未读计数、过期清理。两者解耦后可以独立扩容——比如离线消息量大时只扩 OfflineService，不影响发送链路）
- **离线消息和主消息表 im_message 是什么关系？** （答：im_message 存储所有消息（不管在线离线），是消息的"真相源"。offline_message 是为离线用户准备的"投递副本"，带有接收者维度的索引，方便按 userId+sessionId 快速查询。用户拉取时优先走 offline_message，也可以通过 /sync 接口从 im_message 按 seq 补齐）
- **如果一条消息既存了 im_message 又存了 offline_message，是不是冗余了？** （答：是的，这是有意的冗余设计。im_message 按 sessionId+seq 索引，适合"拉这个会话最近的消息"。offline_message 按 receiverId+sessionId+seq 索引，适合"拉这个用户的所有离线消息"。群聊场景下，一条群消息在 im_message 只存 1 条，但在 offline_message 可能给每个离线成员各存 1 条）

二、热冷分层存储方案深挖

2.1 整体架构

```
写入（MQ 消费时）:
  storeOfflineMessage()
    ├── storeToRedis()   → 热数据层, 3天 TTL
    └── storeToMySQL()   → 冷数据层, expire_at=now+7天

读取（客户端拉取时）:
  pullSingleSession()
    ├── isRedisHotRangeCovered() ?
    │    ├── YES → pullFromRedisForSession() → 直接返回
    │    └── NO  → pullFromMySQLBySession()  → 兜底返回
```

| 层级 | 存储介质 | 数据结构 | 保留时间 | 适用场景 |
|------|---------|---------|---------|---------|
| 热数据 | Redis | ZSet | 3 天 | 刚下线的用户，很快就会回来拉取 |
| 冷数据 | MySQL | offline_message 表 | 7 天 | 长期离线用户，或热数据过期后的兜底 |

可能被问到的问题：

- **为什么热数据用 3 天、冷数据用 7 天？** （答：IM 场景下大部分用户离线时间 < 3 天——手机锁屏、网络切换、睡觉等。3 天热数据覆盖了 90%+ 的离线补拉场景。7 天冷数据是最终兜底，超过 7 天的消息走历史消息查询，不走离线补拉）
- **为什么热数据用 Redis ZSet 而不是 List？** （答：ZSet 支持按 score（seq）范围查询——ZRANGEBYSCORE(lastSeq, +∞)，天然支持增量拉取。List 只能按下标范围查，做增量拉取需要额外维护游标，且删除中间元素效率低）
- **为什么不只用 Redis？** （答：1) Redis 内存贵，存 7 天的消息内存开销大；2) Redis 重启可能丢数据（取决于持久化策略），MySQL 是最终保障；3) 冷数据查询频率低，MySQL 足够支撑）
- **为什么不只用 MySQL？** （答：MySQL 查询延迟高于 Redis。刚下线的用户重连时需要快速拉取离线消息，Redis ZSet 的 ZRANGEBYSCORE 是 O(log(N)+M) 复杂度，比 MySQL 走 B+ 树索引快一个数量级）

2.2 Redis 热数据写入（OfflineMessageServiceImpl.java:129-142）

```java
private void storeToRedis(Long receiverId, AppMessage appMessage) {
    String zsetKey = "im:offline:" + receiverId + ":" + sessionId;
    String sessionsKey = "im:offline:sessions:" + receiverId;

    String rawMessage = JSON.toJSONString(appMessage);
    double score = appMessage.getSeq() != null ? appMessage.getSeq().doubleValue() : System.currentTimeMillis();

    redisTemplate.opsForZSet().add(zsetKey, rawMessage, score);
    redisTemplate.expire(zsetKey, Duration.ofDays(3));

    redisTemplate.opsForSet().add(sessionsKey, sessionId);
    redisTemplate.expire(sessionsKey, Duration.ofDays(3));
}
```

Redis Key 设计：
| Key | 类型 | Score/Value | TTL | 用途 |
|-----|------|-------------|-----|------|
| `im:offline:{userId}:{sessionId}` | ZSet | score=seq, member=消息JSON | 3天 | 按 seq 有序存储离线消息 |
| `im:offline:sessions:{userId}` | Set | sessionId | 3天 | 记录哪些会话有离线消息 |
| `im:unread:{userId}` | Hash | field=sessionId, value=未读数 | 无TTL | 未读计数 |
| `im:pull:limit:{userId}` | String | 拉取次数 | 1秒 | 拉取限流 |
| `im:offline:notify:{userId}` | List | 通知JSON | 7天 | 离线通知 |

可能被问到的问题：

- **ZSet 的 score 用 seq 而不是时间戳，为什么？** （答：seq 是会话级单调递增的，天然支持增量拉取——客户端传 lastSeq，服务端用 ZRANGEBYSCORE(lastSeq, +∞) 精确拉取新消息。用时间戳的话，同一秒内多条消息的时间戳可能相同，无法精确分页）
- **ZSet 的 member 直接存完整消息 JSON，会不会太大？** （答：单条消息 JSON 约 200~500 字节。假设一个用户离线期间收到 1000 条消息，10 个会话，每个 ZSet 约 100 条 × 500B = 50KB，10 个 ZSet 共 500KB。对 Redis 来说不大。但如果消息体含图片 base64 之类的大字段，需要考虑只存 messageId 引用）
- **expire(zsetKey, 3天) 是整个 ZSet 的过期时间，如果 3 天内又收到新消息呢？** （答：每次 add 后都会重新 expire，等于"续期"。只有连续 3 天没有新消息，ZSet 才会过期。这是合理的——如果会话一直有消息进来，热数据一直有效）
- **`im:offline:sessions:{userId}` 这个 Set 的作用？** （答：跨会话拉取时，先查这个 Set 知道哪些会话有离线消息，然后逐个会话拉取。避免盲目扫描所有会话的 ZSet）

2.3 MySQL 冷数据写入（OfflineMessageServiceImpl.java:198-221）

```java
private void storeToMySQL(Long receiverId, AppMessage appMessage) {
    // 幂等检查：messageId + receiverId 是否已存在
    Long existed = offlineMessageMapper.countByReceiverAndMessageId(receiverId, appMessage.getMessageId());
    if (existed != null && existed > 0) {
        return;  // 跳过重复
    }

    OfflineMessage offlineMessage = new OfflineMessage();
    offlineMessage.setMessageId(appMessage.getMessageId());
    offlineMessage.setReceiverId(receiverId);
    offlineMessage.setSessionId(appMessage.getSessionId());
    offlineMessage.setSeq(appMessage.getSeq());
    offlineMessage.setContent(JSON.toJSONString(appMessage));  // 存完整 JSON
    offlineMessage.setExpireAt(LocalDateTime.now().plusDays(7)); // 7 天过期
    offlineMessage.setStatus(0);  // 0=未读
    offlineMessageMapper.insert(offlineMessage);
}
```

可能被问到的问题：

- **先 COUNT 再 INSERT 不是原子的，有并发问题吗？** （答：有。两个消费线程同时 COUNT=0，然后都 INSERT，第二个会因为唯一索引冲突报 DuplicateKeyException。但代码中没有 catch 这个异常。改进：用 INSERT IGNORE 或 INSERT ON DUPLICATE KEY UPDATE 代替 COUNT+INSERT）
- **content 字段存完整 AppMessage JSON，和 im_message.content 存消息体不同。为什么？** （答：offline_message.content 存的是整个 AppMessage 序列化后的 JSON，包含 userName、avatar、sessionName 等前端展示信息。这样客户端拉取离线消息时直接反序列化就能渲染 UI，不需要再查用户信息。im_message.content 只存消息体 body）
- **expireAt = now + 7 天，这个字段怎么用的？** （答：定时清理任务 OfflineMessageCleanTask 每天凌晨 2 点扫描 expire_at < now 的记录并删除。也可以在查询时 WHERE expire_at > NOW() 过滤，但当前代码没有加这个条件——依赖定时任务清理）

2.4 热数据判断逻辑（OfflineMessageServiceImpl.java:255-270）

```java
private boolean isRedisHotRangeCovered(Long userId, String sessionId, long lastSeq) {
    Long minHotSeq = getMinHotSeq(userId, sessionId);
    return minHotSeq != null && lastSeq >= minHotSeq - 1;
}

private Long getMinHotSeq(Long userId, String sessionId) {
    // ZRANGEWITHSCORES key 0 0 → 取 ZSet 中 score 最小的元素
    Set<TypedTuple<String>> tuples = redisTemplate.opsForZSet()
        .rangeWithScores(buildZSetKey(userId, sessionId), 0, 0);
    // 返回最小 score（即最小 seq）
    return score.longValue();
}
```

判断逻辑：
- 客户端传入 lastSeq（已知的最大 seq）
- 如果 Redis ZSet 中最小的 seq <= lastSeq + 1，说明 Redis 的热数据能覆盖客户端需要的增量范围
- 否则说明客户端太久没拉了，热数据已经不够用，要走 MySQL

可能被问到的问题：

- **举个具体例子说明热数据覆盖判断？**
  ```
  场景1: 客户端 lastSeq=100, Redis ZSet 最小 seq=98
    → 98 <= 100+1=101 → 覆盖 → 从 Redis 拉 seq>100 的消息
  
  场景2: 客户端 lastSeq=50, Redis ZSet 最小 seq=98
    → 98 > 50+1=51 → 不覆盖 → 说明 seq=51~97 的消息已经不在 Redis 了
    → 走 MySQL
  
  场景3: Redis ZSet 为空（key 不存在或已过期）
    → minHotSeq=null → 不覆盖 → 走 MySQL
  ```
- **这个判断有漏洞吗？** （答：有边界情况——如果 Redis ZSet 中的消息不连续（某条消息写入失败），minHotSeq 可能很小但中间有空洞。判定为"覆盖"后从 Redis 拉，可能漏掉中间的消息。但这种情况概率很低，且 MySQL 是兜底）

三、离线消息拉取方案

3.1 单会话拉取（OfflineMessageServiceImpl.java:67-76）

```
POST /api/offline/pull
{
  "sessionId": "sess_xxx",
  "lastSeq": 100,
  "pageSize": 50

```

流程：
1. checkPullLimit(userId) — 限流检查
2. isRedisHotRangeCovered() — 判断热数据是否覆盖

3.2 Redis 拉取（OfflineMessageServiceImpl.java:144-169）

```java
private List<AppMessage> pullFromRedisForSession(Long userId, String sessionId, Long lastSeq, int limit) {
    String zsetKey = "im:offline:" + userId + ":" + sessionId;
    
    // ZRANGEBYSCORE key (lastSeq+ε) +inf LIMIT 0 limit
    Set<String> rawMessages = redisTemplate.opsForZSet()
        .rangeByScore(zsetKey, lastSeq + SCORE_EPSILON, Double.POSITIVE_INFINITY, 0, limit);
    
    List<AppMessage> result = new ArrayList<>();
    for (String raw : rawMessages) {
        AppMessage message = parseOfflineAppMessage(raw);
        if (isValidOfflineAppMessage(message)) {
            result.add(message);
        }
        // 🔑 拉取后删除——"消费即删除"语义
        redisTemplate.opsForZSet().remove(zsetKey, raw);
    }
    
    // 如果 ZSet 为空，清除 sessions Set 中的记录
    Long remaining = redisTemplate.opsForZSet().zCard(zsetKey);
    if (remaining != null && remaining == 0) {
        redisTemplate.opsForSet().remove("im:offline:sessions:" + userId, sessionId);
    }
    return result;
}
```

可能被问到的问题：

- **拉取后就删除（remove），如果客户端没有成功处理怎么办？** （答：这是"消费即删除"模型的风险。如果客户端拉到了消息但渲染前 APP 崩溃，消息在 Redis 中已删除。补救方式：1) MySQL 冷数据还在，客户端下次拉取时走 MySQL 兜底；2) 客户端也可以通过 /sync 接口从 im_message 按 seq 补齐。所以 Redis 层的删除是安全的——MySQL 和 im_message 是双重兜底）
- **SCORE_EPSILON = 0.000001 是干什么的？** （答：ZRANGEBYSCORE 默认是闭区间 [min, max]。加一个微小的 epsilon 变成开区间效果——排除 score 等于 lastSeq 的消息，只返回 seq > lastSeq 的消息。等价于 ZRANGEBYSCORE key (lastSeq +inf）
- **rangeByScore 和 remove 不是原子的，高并发下会不会重复拉取？** （答：会。如果同一用户在两个设备同时拉取同一会话的离线消息，两个请求都 rangeByScore 拿到同一批消息，然后都 remove。但 remove 是幂等的——第二次 remove 返回 0 个被删除。客户端按 messageId 去重即可）

3.3 MySQL 拉取（OfflineMessageServiceImpl.java:98-108 + OfflineMessageMapper.java:14-26）

```sql
SELECT * FROM offline_message
WHERE receiver_id = #{receiverId}
  AND session_id = #{sessionId}
  AND seq > #{lastSeq}
  AND status = 0
ORDER BY seq ASC
LIMIT #{limit}
```

- **为什么要 status = 0？** （答：0=未读。已读消息（status=1）不需要再拉取。但当前代码中 "标记已读" 是直接清零未读计数，没有更新 offline_message.status。这个 status 字段实际上没有被更新为 1 的地方——只有定时清理任务才用它过滤）

3.4 跨会话拉取（OfflineMessageServiceImpl.java:78-96）

```
POST /api/offline/pull

  "sessionSeqMap": {
    "sess_1": 100,
    "sess_2": 50,
    "sess_3": 200
  },
  "pageSize": 50
```

流程：
1. 遍历 sessionSeqMap 中每个会话
2. 每个会话分别判断热/冷 → 拉取


可能被问到的问题：

- **跨会话拉取时，如果用户有 100 个会话，每个会话都要 rangeByScore / MySQL 查询一次，性能怎么样？** （答：确实有性能问题。100 个会话 = 100 次 Redis ZRANGEBYSCORE + 100 次可能的 MySQL 查询。优化方向：1) 先查 im:offline:sessions:{userId} 这个 Set，只拉有离线消息的会话；2) Redis Pipeline 批量查询；3) 限制单次拉取的会话数）
- **合并后按 createdAt 排序，如果不同会话的 createdAt 格式不一致怎么办？** （答：AppMessage.createdAt 是格式化字符串 "yyyy-MM-dd HH:mm:ss"，String 自然排序恰好是时间顺序。但如果时间格式不统一，排序会出错）

4.1 写入（OfflineMessageListener → UnreadCountServiceImpl.java:62-72）

每条离线消息入库时：

incrementUnreadCount SQL（UnreadCountMapper.java:12-18）：

```sql
VALUES (#{userId}, #{sessionId}, 1, #{messageId}, #{messageTime}, NOW())

  last_message_time = #{messageTime},
```

4.2 读取（UnreadCountServiceImpl.java:30-58）

```java
public Map<String, Integer> getUnreadCount(Long userId) {
    String key = "im:unread:" + userId;
    
    // 1. 先查 Redis Hash
    Map<Object, Object> redisMap = redisTemplate.opsForHash().entries(key);
    if (redisMap != null && !redisMap.isEmpty()) {
        return redisMap 转换为 Map<String, Integer>;
    }
    
    // 2. Redis 未命中 → 查 MySQL → 回写 Redis
    List<UnreadCount> unreadCounts = unreadCountMapper.selectList(WHERE user_id = userId);
    for (UnreadCount uc : unreadCounts) {
        redisTemplate.opsForHash().put(key, uc.getSessionId(), String.valueOf(uc.getUnreadCount()));
    }
    return result;
}
```

4.3 清零（UnreadCountServiceImpl.java:76-89）

```java
// markAsRead / clearUnread
redisTemplate.opsForHash().delete(key, sessionId);       // Redis 删除该会话的未读数
unreadCountMapper.clearUnreadCount(userId, sessionId);    // MySQL SET unread_count = 0
```

可能被问到的问题：

- **Redis Hash 和 MySQL 双写，一致性怎么保证？** （答：当前是"先写 Redis 再写 MySQL"。如果 Redis 写成功但 MySQL 写失败，未读数就不一致。反过来也一样。没有事务保证。可接受的原因：未读数是辅助信息，不影响消息本身的可靠性。最坏情况是未读数不准，用户打开会话后清零就对了）
- **HINCRBY 是原子操作吗？** （答：是。Redis 单线程模型下 HINCRBY 天然原子。多条离线消息并发入库时不会丢计数）
- **INSERT ON DUPLICATE KEY UPDATE 的主键/唯一索引是什么？** （答：应该是 (user_id, session_id) 联合唯一索引。第一次插入 unread_count=1，后续更新 +1）
- **decrementUnreadCount 方法用了 GREATEST(unread_count - #{count}, 0)，为什么？** （答：防止未读数变成负数。可能场景：Redis 中未读数 = 5，但 MySQL 中因为某些原因未读数 = 3，减 5 后变成 -2。GREATEST 保底为 0）
- **未读数有没有上限？如果用户长期离线，某个群有 10000 条未读？** （答：当前没有上限。客户端通常显示 "99+" 或 "999+"。服务端可以考虑 cap 到某个值，但实际影响不大——只是一个数字）

5.1 OfflineNotifyListener（OfflineNotifyListener.java:18-57）

```
Topic: IM_NOTIFY, Tag: OFFLINE
消费: 好友申请通知、朋友圈通知等非聊天类消息
存储: Redis List im:offline:notify:{userId}
```


客户端拉取（OfflineMessageController.java:67-78）：
```java
@PostMapping("/notifications")
public Result<List<String>> pullOfflineNotifications(Long userId) {
    String key = "im:offline:notify:" + userId;
    List<String> notifications = redisTemplate.opsForList().range(key, 0, -1);
    if (notifications != null && !notifications.isEmpty()) {
        redisTemplate.delete(key);  // 拉完就删
    }
    return notifications;
}
```

可能被问到的问题：

- **为什么离线通知只用 Redis List，不像聊天消息那样双写 MySQL？** （答：通知类消息（好友申请、朋友圈提醒）的优先级低于聊天消息。丢了不影响核心体验——用户打开好友列表或朋友圈时还会重新加载。Redis List 够用，不需要 MySQL 持久化）
- **trim(key, -200, -1) 是什么意思？** （答：保留 List 尾部的 200 条（即最新的 200 条）。如果用户长期离线，通知堆积超过 200 条，老的通知会被丢弃。这是合理的——200 条未读通知已经足够了）
- **range(key, 0, -1) 一次取出所有通知，如果有 200 条会不会有性能问题？** （答：200 条 × 每条约 500 字节 = 100KB，一次网络传输没问题。但如果通知体很大，可以分页拉取）
- **拉完就 delete(key)，如果客户端处理失败呢？** （答：通知类消息允许丢失。和离线聊天消息的"消费即删除"类似，但通知没有 MySQL 兜底。这是设计取舍——简单 > 可靠）

六、限流方案

6.1 拉取限流（OfflineMessageServiceImpl.java:241-253）

```java
private boolean checkPullLimit(Long userId) {
    String key = "im:pull:limit:" + userId;
    Long count = redisTemplate.opsForValue().increment(key);  // INCR 原子自增
    if (count == 1) {
        redisTemplate.expire(key, 1, TimeUnit.SECONDS);       // 第一次设置 1 秒过期
    }
    return count <= PULL_LIMIT_PER_SECOND;  // 每秒最多 10 次
}
```

可能被问到的问题：

- **INCR + EXPIRE 不是原子的，有什么问题？** （答：经典的 Race Condition。如果第一次 INCR 后进程崩了，没来得及 EXPIRE，key 就永远不会过期，该用户就被永久限流。修复方案：用 Lua 脚本把 INCR + EXPIRE 合为原子操作，或者用 SET key 1 EX 1 NX + INCR 的组合）
- **count == 1 判断设置过期有什么边界问题？** （答：如果上一个 1 秒窗口的 key 刚好在 INCR 前一瞬间过期，count 又变成 1，重新设置 1 秒过期——这是正常行为。但如果网络延迟导致 expire 命令比 incr 晚很多执行，窗口长度可能不准确）
- **每秒 10 次拉取是针对什么场景？** （答：防止恶意客户端或 bug 导致的高频轮询。正常客户端重连后拉一次离线消息就够了，不需要每秒拉 10 次）

七、定时清理任务

7.1 OfflineMessageCleanTask（OfflineMessageCleanTask.java:27-90）

```
@Scheduled(cron = "0 0 2 * * ?")  — 每天凌晨 2 点
分布式锁: SETNX im:task:offline-message-clean, TTL=10分钟

清理逻辑：
1. 删除过期消息: WHERE expire_at < now - 7天, LIMIT 1000/批
2. 删除已读消息: WHERE status = 1 AND created_at < now - 3天, LIMIT 1000/批
3. 每批删完 sleep(1000ms) 防止锁表
```

可能被问到的问题：

- **为什么用 SETNX 做分布式锁而不是 Redisson？** （答：简单场景下 SETNX + TTL 够用。Redisson 的 RLock 有 Watchdog 自动续期，适合执行时间不确定的长任务。这里清理任务是分批处理 + 10 分钟 TTL，足够安全。但严格来说，如果清理任务执行超过 10 分钟，锁会被其他实例获取，两个实例同时清理——不过清理是幂等的，不会出问题）
- **逐条 deleteById 效率很低，为什么不用 DELETE WHERE ... LIMIT 1000？** （答：确实可以优化。当前实现是 selectList + 逐条 deleteById，N 次 DELETE 语句。优化为 DELETE WHERE id IN (...) 批量删除，或直接 DELETE FROM offline_message WHERE expire_at < ? LIMIT 1000）
- **sleep(1000) 在锁的 TTL 内安全吗？** （答：1000 条/批 × sleep 1s/批。如果有 10000 条待清理，需要 10 批 × 1s = 10s。加上 SQL 执行时间，总时长可能几十秒。10 分钟 TTL 足够。但如果数据量非常大（百万级），需要加大 TTL 或改用 Redisson RLock）
- **Redis 热数据不需要清理？** （答：不需要。ZSet 设了 3 天 TTL，Redis 自动过期删除。Session Set 也是 3 天 TTL）

九、RocketMQ 消费与触发链路

9.1 消息回流路径

```
MessageStoreListener (Messaging Service)
  → 查 Redis im:route:{userId}
  → 发现用户离线（route 不存在或 = "OFFLINE"）
  → syncSendOrderly("IM_CHAT:OFFLINE", packet, sessionId)
      ↓
OfflineMessageListener (OfflineService)
  [topic=IM_CHAT, tag=OFFLINE, group=im-offline-message-group]
  → storeOfflineMessage + incrementUnreadCount
```

```
NettyPushMessageListener (RTC Service)
  → channel.writeAndFlush() 推送失败（用户在推送瞬间断线）
  → syncSend("IM_CHAT:OFFLINE", packet, 3000ms)
      ↓
OfflineMessageListener (OfflineService)
  → 同上
```

可能被问到的问题：

- **消息有两个入口进 OFFLINE：1) MessageStoreListener 路由发现离线 2) NettyPushMessageListener 推送失败。会不会重复？** （答：会。用户可能在 MessageStoreListener 查路由时在线，但推到 Netty 节点后刚好断线，导致推送失败回流 OFFLINE。这时 im_message 已经存了，MessageStoreListener 那边不会再发 OFFLINE（因为查路由时是在线的），但 NettyPushMessageListener 会发一条 OFFLINE。所以同一条消息可能走两条路径，storeToMySQL 的幂等检查（countByReceiverAndMessageId）防止重复落库）
- **IM_CHAT:OFFLINE 的消费端是 CONCURRENTLY 模式，消息处理顺序不保证？** （答：是的。同一会话的两条离线消息可能被两个消费线程并发处理。但无所谓——每条消息独立写入 Redis ZSet（score=seq 保证有序）和 MySQL（seq 字段保证查询时有序）。写入顺序不影响读取顺序）
- **如果 OfflineService 本身宕机了，MQ 消息堆积怎么办？** （答：RocketMQ 消息持久化在 Broker。OfflineService 恢复后自动消费堆积的消息。消息不会丢。但用户在这段时间内拉取离线消息会拉不到——直到 OfflineService 恢复并处理完堆积）

十、客户端重连离线消息补齐流程（端到端）

10.1 完整补齐流程

```
客户端检测到断线
  → 指数退避 + 随机抖动重连 WebSocket
  → 连接成功

步骤1: 获取所有会话的 lastAckSeq
  GET /api/message/sessions/ackSeqMap (Messaging Service)
  → {session1: 100, session2: 50, session3: 200}

步骤2: 获取每个会话的当前最大 seq
  GET /api/message/maxSeq?sessionId=session1 (Messaging Service)
  → session1 当前 seq=120

步骤3: 如果 lastAckSeq < maxSeq，有新消息需要补齐
  方式A（走离线服务）:
    POST /api/offline/pull {sessionId: "session1", lastSeq: 100, pageSize: 50}
    → 从 Redis ZSet / MySQL 拉取 seq=101~120 的消息
  
  方式B（走消息同步）:
    GET /api/message/sync?sessionId=session1&beginSeq=100&endSeq=120
    → 从 im_message 主表按 seq 范围查询

步骤4: 获取未读计数
  GET /api/offline/unread
  → {session1: 20, session2: 5}

步骤5: 渲染 UI，发送 ACK
  POST /api/message/ack?sessionId=session1&seq=120
```

可能被问到的问题：

- **方式 A（/offline/pull）和方式 B（/message/sync）有什么区别？什么时候用哪个？** （答：
  - /offline/pull：从 offline_message 表按 receiverId 维度查询，带 Redis 热数据加速。适合拉取"发给我的离线消息"
  - /message/sync：从 im_message 主表按 sessionId+seq 维度查询，返回 tombstone 标记空洞。适合拉取"这个会话的历史消息"
  - 客户端可以二选一：如果有离线服务，优先用 /offline/pull（更快）；否则用 /message/sync（通用兜底）
  - 两者数据源不同但内容一致）
- **补齐流程要发多少次 HTTP 请求？** （答：ackSeqMap 1 次 + 每个会话 maxSeq 1 次 + pull/sync 1 次 + unread 1 次 + ack N 次。如果有 10 个会话，约 20+ 次请求。优化：合并接口——比如 ackSeqMap 可以同时返回 maxSeq，减少一半请求）

十二、Redis 数据结构八股

12.1 ZSet（有序集合）

底层实现：**ziplist（数据量小）+ skiplist + hashtable（数据量大）**

| 操作 | 命令 | 时间复杂度 | 你的使用场景 |
|------|------|-----------|------------|
| 添加 | ZADD | O(log N) | storeToRedis: 写入离线消息 |
| 按 score 范围查 | ZRANGEBYSCORE | O(log N + M) | pullFromRedisForSession: 增量拉取 |
| 删除指定成员 | ZREM | O(log N) | 拉取后删除 |
| 获取最小 score | ZRANGEWITHSCORES 0 0 | O(log N + 1) | isRedisHotRangeCovered: 判断热数据覆盖 |
| 获取集合大小 | ZCARD | O(1) | 判断是否清空 sessions Set |
| 设置过期时间 | EXPIRE | O(1) | 3 天 TTL |

可能被问到的问题：

- **skiplist 是什么？为什么 Redis ZSet 用跳表而不是红黑树？** （答：跳表是多层链表，查找 O(log N)。Redis 选跳表的原因：1) 实现简单；2) 范围查询高效——skiplist 的有序链表天然支持范围遍历，红黑树做范围查询需要中序遍历；3) 并发友好——跳表的局部修改不影响全局结构）
- **ZSet 的 member 是完整消息 JSON（约 500 字节），有什么影响？** （答：member 参与 hashtable 的 key 比较和 skiplist 的节点存储。大 member 会增加内存占用和比较开销。如果消息体很大，可以改为只存 messageId，拉取时再从 MySQL 查完整内容）

12.2 Hash（哈希表）

| 操作 | 命令 | 时间复杂度 | 你的使用场景 |
|------|------|-----------|------------|
| 自增 | HINCRBY | O(1) | incrementUnreadCount: 未读数 +1 |
| 获取所有 | HGETALL | O(N) | getUnreadCount: 获取所有会话未读数 |
| 删除字段 | HDEL | O(1) | clearUnread: 清零未读数 |

12.3 List（列表）

| 操作 | 命令 | 时间复杂度 | 你的使用场景 |

| 右插入 | RPUSH | O(1) | 离线通知存储 |
| 截取 | LTRIM | O(N) | 保留最新 200 条通知 |

十七、生产环境优化与架构演进

面试官可能在以上基础上继续追问：

- **离线消息量爆增怎么办？** （答：1) offline_message 按 receiver_id 分表；2) Redis ZSet 超过一定大小后只存最近 N 条，更早的走 MySQL；3) 冷数据归档到 TiKV/HBase）
- **如果 Redis 宕机，离线消息只存了 MySQL，拉取延迟会飙升吗？** （答：会。可以用 Redis Sentinel/Cluster 做高可用。或者在 MySQL 层加 READ REPLICA 做读写分离，分散查询压力）
- **群聊 500 人的群，一条消息产生 500 条 offline_message，写放大严重吗？** （答：严重。每条约 500B × 500 = 250KB 写入 MySQL + Redis。高频群聊下 offline_message 增长极快。优化：群聊离线消息不按接收者写扩散，改为存一条 + 群成员 bitmap 标记已读/未读。但实现复杂度高，需要权衡）
- **未读计数和离线消息数量不一致怎么办？** （答：incrementUnreadCount 和 storeOfflineMessage 没有事务保证。如果 store 成功但 increment 失败，消息存了但未读数没加。客户端打开会话后可以通过实际消息数量校准。也可以定时对账：COUNT(*) WHERE receiver_id=? AND session_id=? AND status=0 和 unread_count 比较）
- **用户在手机和电脑同时在线，手机收到了消息标记已读，电脑的未读数怎么同步？** （答：markAsRead 时通过 WebSocket 推送一条"已读同步通知"给同一用户的其他设备。当前架构只支持单设备在线，不存在这个问题。如果要支持多设备，需要额外的已读同步机制）
- **离线消息的推送通知（APP Push）怎么做？** （答：当前架构没有 APP Push。如果要做，OfflineMessageListener 消费时额外调用 APNs（iOS）/ FCM（Android）推送一条通知栏消息，提示用户有新消息）
- **如何保证离线消息不丢不重？** （答：
  - 不丢：MQ 至少一次投递 + MySQL 持久化。最坏情况下 Redis 丢了还有 MySQL，MySQL 丢了还有 im_message 主表 /sync 补齐
  - 不重：storeToMySQL 幂等检查（countByReceiverAndMessageId）+ 客户端按 messageId 去重
  - 残留风险：storeToMySQL 的 COUNT + INSERT 非原子，极端并发下可能 DuplicateKeyException 但没被 catch——需要加 catch 或改为 INSERT IGNORE）
