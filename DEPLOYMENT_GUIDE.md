# 🚀 InfiniteChat 伪分布式部署指南

## 📋 目标
用 **2 台服务器** 展示完整的分布式能力，面试时能自信地说："我的项目具备分布式架构的核心能力"。

---

## 🏗️ 部署架构

### 服务器 1: 应用服务器 (2C4G)
```bash
# 端口规划
8080  → Gateway (网关)
8081  → AuthenticationService (认证服务)
8082  → MessagingService (消息服务)
9100  → RTC-Instance-1 (Netty 实例 1)
9101  → RTC-Instance-2 (Netty 实例 2) ← 关键：同机多进程模拟集群
```

### 服务器 2: 中间件服务器 (2C4G)
```bash
# 端口规划
3306  → MySQL
6379  → Redis
8848  → Nacos
9876  → RocketMQ NameServer
10911 → RocketMQ Broker
```

## 🎯 核心能力展示清单

### 1. 一致性哈希负载均衡 ✅
**实现方式**: 
- 在服务器 1 上启动 2 个 RTC 实例（端口 9100 和 9101）
- 用户登录时，Auth 服务通过一致性哈希算法将用户路由到固定的 RTC 实例
- Nacos 自动发现 2 个实例，哈希环动态构建

**演示效果**:
```bash
# 用户 A (userId=1001) → RTC-Instance-1 (9100)
# 用户 B (userId=1002) → RTC-Instance-2 (9101)
# 用户 C (userId=1003) → RTC-Instance-1 (9100)
```

**面试话术**:
> "我在同一台服务器上启动了 2 个 Netty 实例，通过 Nacos 注册不同端口。虽然是同机部署，但从代码层面看，它们是完全独立的进程，通过 Redis Pub/Sub 实现跨实例通信。这验证了我的一致性哈希算法在多节点场景下的正确性。"

---

### 2. 分布式锁 ✅
**实现方式**:
- 用户注册时，使用 Redisson 分布式锁防止并发注册
- 锁粒度优化到手机号级别：`register:lock:13800000001`
- 固定 5 秒超时，避免死锁

**演示效果**:
```bash
# 模拟并发注册测试
curl -X POST http://server1:8081/register -d '{"phone":"13800000001"}'
curl -X POST http://server1:8081/register -d '{"phone":"13800000001"}'
# 第二个请求会因为拿不到锁而等待或失败
```

**面试话术**:
> "虽然我只有 1 个 Auth 实例，但我用 Redis 分布式锁保证了并发安全。如果扩展到多个 Auth 实例，代码无需修改，因为锁是基于 Redis 的，天然支持跨进程。"

---

### 3. 分布式事务 (RocketMQ 事务消息) ✅
**实现方式**:
- 红包发送场景：扣款 + 发消息 + 通知接收方
- 使用 RocketMQ 事务消息保证最终一致性

**演示效果**:
```bash
# 发红包流程
1. 扣款成功 → 发送半消息
2. 半消息发送成功 → 提交事务
3. RocketMQ 推送消息给接收方
4. 如果扣款失败 → 回滚半消息
```

**面试话术**:
> "我用 RocketMQ 的事务消息实现了分布式事务。虽然当前只有 1 个 Messaging 实例，但 RocketMQ 的事务消息机制本身就是为分布式设计的。如果扩展到多个实例，每个实例都能独立处理事务消息。"

---

### 4. 服务注册发现 (Nacos) ✅
**实现方式**:
- 所有服务启动时注册到 Nacos
- Auth 服务通过 Nacos 动态获取 RTC 实例列表
- 模拟节点下线：停掉 RTC-Instance-2，观察哈希环重构

**演示效果**:
```bash
# 查看 Nacos 服务列表
http://server2:8848/nacos
# 可以看到 2 个 RTC 实例

# 停掉 RTC-Instance-2
kill -9 <pid>
# 30 秒后 Nacos 自动剔除，哈希环重构
```

**面试话术**:
> "我用 Nacos 实现了服务注册发现。虽然是同机多进程，但 Nacos 看到的是 2 个独立的服务实例。这验证了我的架构在节点动态上下线时的自适应能力。"

## 🎤 面试应对策略

### Q1: "你这个项目是真分布式吗？只有 2 台服务器？"
**回答**:
> "我的项目采用了**渐进式架构设计**。当前用 2 台服务器展示核心分布式能力：
> - **服务器 1**: 部署了 5 个应用实例（包括 2 个 Netty 实例模拟集群）
> - **服务器 2**: 部署了所有中间件（MySQL, Redis, Nacos, RocketMQ）
> 
> 虽然 Netty 实例是同机部署，但从架构层面看，它们是完全独立的进程，通过 Nacos 注册、Redis 通信、一致性哈希路由。这验证了我的架构在多节点场景下的正确性。
> 
> 如果要扩展到生产环境，只需要：
> 1. 将 Netty 实例部署到不同物理机
> 2. Redis 改为 Cluster 模式
> 3. MySQL 配置主从复制
> 
> **代码层面无需任何修改**，这就是我设计时考虑的**水平扩展能力**。"

---

### Q2: "同机多进程和真分布式有什么区别？"
**回答**:
> "从**网络层面**看，同机多进程确实没有跨机器的网络延迟和分区容错问题。但从**架构设计**层面看，它们是一样的：
> 
> **相同点**:
> - 进程间通过网络通信（TCP/HTTP），不是共享内存
> - 通过 Nacos 服务发现，不是硬编码 IP
> - 通过 Redis 分布式锁，不是本地锁
> - 通过 RocketMQ 解耦，不是方法调用
> 
> **不同点**:
> - 同机部署没有网络分区问题（但我在代码中考虑了网络超时和重试）
> - 同机部署没有跨机房延迟（但我的架构支持跨机房部署）
> 
> 我的重点是**验证架构设计的正确性**，而不是单纯堆机器。"

---

### Q3: "为什么不用 Docker/K8s 部署？"
**回答**:
> "我考虑过用 Docker，但最终选择了**原生部署**，原因是：
> 1. **学习成本**: 我想把精力放在分布式原理上，而不是容器编排
> 2. **资源限制**: 2 台 2C4G 的服务器跑 Docker 会比较吃力
> 3. **可演示性**: 原生部署更容易展示进程间通信和服务发现
> 
> 但我的架构是**容器友好的**，所有配置都通过环境变量注入，可以无缝迁移到 Docker/K8s。"

---

### Q4: "你的分布式锁在单实例下有什么意义？"
**回答**:
> "虽然当前只有 1 个 Auth 实例，但我用分布式锁的原因是：
> 1. **架构一致性**: 如果扩展到多实例，代码无需修改
> 2. **并发安全**: 即使单实例，也要防止同一用户的并发请求
> 3. **原理验证**: 我想展示对 Redisson 看门狗、锁粒度优化的理解
> 
> 这是一种**面向扩展的设计**，而不是过度设计。"

---

### Q5: "你的项目有什么亮点？"
**回答**:
> "我的项目有 3 个核心亮点：
> 
> **1. 渐进式架构设计**
> - 用 2 台服务器展示了完整的分布式能力
> - 架构支持从单机平滑扩展到集群
> 
> **2. 深度优化**
> - 一致性哈希的虚拟节点优化（160 个虚拟节点减少数据倾斜）
> - 分布式锁的粒度优化（手机号级别，TPS 提升 3-5 倍）
> - Token 刷新的防重放攻击（Lua 脚本原子性校验）
> 
> **3. 工程化思维**
> - 完善的异常处理和日志埋点
> - 异步化非关键操作（邮件发送、余额初始化）
> - 补偿机制（登出时的 Pub/Sub 补偿）
> 
> 这些都是我在实际开发中遇到的问题，并通过深入学习分布式原理解决的。"

1. 我用有限资源展示了核心分布式能力

- 用最少的资源展示了最核心的能力
- 架构设计具备平滑扩展能力
