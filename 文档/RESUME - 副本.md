# InfiniteChat — 简历项目经历

---

## 项目介绍

基于微服务架构的分布式即时通信系统，涵盖用户认证、消息收发、实时推送、离线消息、红包、朋友圈等完整 IM 业务链路。采用 Spring Cloud 微服务治理（7 个服务节点），RocketMQ 异步解耦，Netty 实现 WebSocket 长连接推送，Redis 多数据结构支撑缓存/路由/限流/分布式锁等核心场景。部署于 2 核 4G（中间件）+ 2 核 8G（业务）双节点环境，经阶梯式压测验证核心链路稳定性。

---

## 核心技术栈

Spring Boot、Spring Cloud Gateway、Netty、RocketMQ、Redis/Redisson、MySQL、MyBatis-Plus、Nacos、OpenFeign、JWT

## 核心功能

- **登录认证服务**：注册阶段通过 Redisson 分布式锁串行化同手机号请求，结合事务完成用户与余额账户初始化；登录阶段签发 AT/RT 双 Token，借助 Redis + Lua 实现 RefreshToken 单次使用与登出统一失效，并在登录成功后基于服务发现与节点负载返回 RTC 直连地址。

1. 概述

这个模块我主要解决的是账号生命周期管理的问题，包括注册防重、登录态续期和登出失效。注册阶段用 Redisson 分布式锁防止同一手机号并发注册导致重复写入；登录阶段采用 AT/RT 双 Token，把高频鉴权和可续签能力分开。

2. 方案层

1、为什么选 Redisson，不只靠数据库唯一索引

A: 我这里其实是两层保护。第一层是 Redisson 分布式锁，按手机号把注册请求串行化，避免同一手机号在极短时间内并发进入注册核心逻辑；第二层是数据库唯一索引，防止极端情况下出现重复数据。
如果只靠数据库唯一索引，最终大概率也不会插入两条相同手机号的数据，但会出现一种情况：多个请求同时通过“查无此人”的校验，然后一起冲到写库阶段，其中一个成功，其他请求在数据库层抛唯一键冲突。这样虽然数据最终没脏，但业务过程不够平滑，错误处理也比较被动。
所以我加分布式锁的目的，是把冲突从数据库层前移到业务层，减少无效写库和异常回滚，让注册语义更稳定。

2、为什么选双 Token，不直接只用 JWT

A: 如果只用一个 JWT，会有两个问题。第一，Token 过期时间如果设置得很短，用户需要频繁重新登录，体验很差；第二，如果把它设置得很长，一旦泄露，风险窗口就会比较大。
所以我把它拆成双 Token：AccessToken 负责高频接口鉴权，过期时间比较短，我这里是 2 小时；RefreshToken 只负责换取新的 AccessToken，过期时间更长，我这里是 7 天。这样用户平时访问只带 AT，只有在 AT 过期时才走 RT 续签。
另外在我的实现里，RefreshToken 不是完全无状态的，而是存 Redis，并通过 Lua 做 compare-and-delete，保证单次使用。也就是说，旧的 RT 用过一次就作废，服务端再签发一组新的 Token，这样能进一步降低 RT 被重复使用的风险。

3、数据怎么流转

- 发送验证码：客户端调用 /api/v1/user/noToken/sms 服务端生成验证码，用 phone -> code 写入 Redis，TTL 5 分钟，然后异步发到测试邮箱模拟短信。
- 注册：客户端调用 /api/v1/user/noToken/register，提交 phone/password/code。服务端在锁外先做三件事：BCrypt.hashpw(password)、雪花算法生成 userId、生成随机昵称。然后按 register:lock:{phone} 获取 Redisson 分布式锁。拿到锁后，在事务里执行：从 Redis 取验证码并校验、查手机号是否已注册、插入 user、插入 user_balance、删除验证码。
- 密码登录：客户端调用 /api/v1/user/noToken/loginPwd。服务端按手机号查用户，用 BCrypt.checkpw 验密，成功后进入 buildUserVOAndAllocateNode，里面会：签发 AT(2h) 和 RT(7d)、把 RT:{userId} 写入 Redis、返回 nettyUrl。
- RefreshToken 刷新：客户端调 /api/v1/user/refreshToken。服务端先从 RT 里解析 userId 再对 Redis 中的 RT:{userId} 执行 Lua：相等则删除旧 RT，允许刷新，不相等则拒绝。成功后重新生成一组新的 AT + RT + nettyUrl。
- 登出：客户端调用 /api/v1/user/logout。服务端先写 jwt:logout:{userId} 时间戳，TTL 2 小时，再执行 Lua：读旧 im:route:{userId}、删除 RT:{userId}、删除 im:route:{userId}、发布踢人消息。这样登出前签发的旧 AT 会在网关/RTC 校验阶段统一失效。

- **红包系统**：发红包在本地事务中完成余额扣减（MySQL 行锁原子 CAS）、创建红包、记录流水，随后预计算金额列表存入 Redis List（普通红包平均分配，拼手气红包二倍均值法）。抢红包采用 RocketMQ 事务消息保证 Redis 预扣与 MySQL 落库的最终一致：半消息发送后在 executeLocalTransaction 中执行四合一 Lua 脚本（SISMEMBER 防重 + DECR 扣库存 + LPOP 弹金额 + SADD 标记 + HSET 暂存），CompletableFuture 同步返回结果；MQ 消费端异步落库（行锁扣减 + 领取记录 + 加余额 + 流水），Broker 回查 Pending Hash 精准补偿。24 小时延迟消息触发过期退款，CAS 保证幂等。

- **联系人关系管理**：好友同意操作使用 Redisson 分布式锁（锁 key 取 min/max 保证双方互斥）+ 事务双向插入，通过 Feign 降级调用 Messaging 创建会话、RTC 推送通知；用户注册通过 MQ 事件驱动异步创建 AI 助手。

## 性能压测与调优

- **登录接口调优**：基于 wrk2 对登录接口进行阶梯式压测（100→200→300 QPS），通过 top + jstack 定位瓶颈为 BCrypt CPU 密集计算（2 核机器 RUNNABLE 线程 90+，CPU 占用 96%，IO Wait 为 0）；将 BCrypt cost 从 10 降至 6，单机吞吐量从 17 QPS 提升至 200 QPS（约 12 倍），P99 从 49s 降至 333ms；确认拐点在 200 QPS，超过后 P50 从 139ms 飙升至 2.16s，为生产环境容量规划提供数据支撑。

- **消息发送链路调优**：针对单聊消息发送接口进行全链路压测。初始 200 QPS 下 P99 达 2.46s，排查发现 RocketMQ 同步投递阻塞 Tomcat 线程池，改为 asyncSendOrderly 异步顺序投递，失败由回调写入本地兜底表（im_msg_failover），延迟大幅下降；随后 500 QPS 压测中发现网关 CPU 占用 53%，通过 jstack 定位到 RateLimitFilter 与 AuthorizeFilter 白名单策略不一致，导致每个请求仍执行 HMAC-SHA512 验签与 Redis 登出时间戳查询，统一白名单后平均延迟从 3.3s 降至 26ms（128 倍）；进一步压测定位到 2 核机器上 7 个 JVM 共享 CPU 为硬件瓶颈（load average 7.57），最终单机 1200 QPS 稳定运行，P99 < 72ms，零错误零超时。

- **红包系统压测**：针对群聊拼手气红包（50 名额/个）进行阶梯式压测（wrk2，10→50→150→300→500→800→1200 QPS），验证四合一 Lua 脚本原子预扣 + RocketMQ 事务消息 + MySQL 异步落库的全链路并发安全性。2 核 8G 单节点下，800 QPS 时 P99 = 51ms、零错误；1200 QPS 出现拐点（P999 从 89ms 飙升至 804ms，瓶颈为 MQ 事务半消息提交尾部延迟与 CPU 调度抖动）。压测后数据一致性验证：零重复领取、零超发、领取总额与红包金额分文不差。


- **Netty 长连接接入层**：基于 Netty 自建 WebSocket 长连接服务，采用 Boss + Worker + Business 线程模型，将 JWT 鉴权前置到握手阶段；在连接建立后通过 Redis 脚本原子注册全局在线路由并维护本地会话映射，结合心跳超时剔除、断连路由清理、Redis Pub/Sub 踢旧连接与优雅停机机制，支撑多节点 RTC 接入。

2. 第二层
- 为什么这里选 Netty 自建 WebSocket 长连接接入层，而不是直接用 Spring WebSocket？
A：netty和spring ws都是基于NIO的框架，但是netty的优势在统一的api，不仅编程更加简单，而且比直接使用 Java 核心 API 有更高的吞吐量、更低的延迟、更低的资源消耗和更少的内存复制
补充：Spring WebSocket 更适合快速实现业务层的 WebSocket 功能，但我这里做的是独立的 RTC 长连接接入层，不只是收发消息，还要自己控制连接建立、握手鉴权、心跳保活、在线路由、断连清理、单设备踢人和优雅停机这些底层能力。相比之下，Netty 对线程模型、Pipeline 和连接生命周期的控制更细，所以我选了 Netty。性能收益是附带结果，但不是我最核心的选型原因。

- 为什么要拆 Boss / Worker / Business 线程模型，而不是只用 Netty 默认的两组线程？
A：我使用的是多线程模型。boss 线程只用于监听连接请求，线程数为 1；worker 用于处理网络连接，线程数为 CPU 核心数；business 用于处理业务逻辑，线程数为 CPU 核心 * 2。之所以这么设计是为了防止业务逻辑造成网络阻塞，当网络请求源源不断时，业务会阻塞后续网络请求的处理，导致整个网络响应变慢，严重者造成网络瘫痪。
补充：我这里把线程模型拆成 Boss、Worker 和 Business 三层。Boss 线程只负责接收连接，数量是 1；Worker 线程负责网络 IO 和编解码，数量按 CPU 核心数配置；Business 线程负责业务处理，数量是 CPU 核心数的 2 倍。这样做的核心目的，是把“网络读写”与“业务逻辑”隔离开，避免业务处理占住 Worker 线程，影响后续连接的读写。因为在我的 RTC 模块里，除了基础的心跳和协议处理外，还会涉及 JWT 鉴权、在线路由维护、断连清理和 Redis 交互，如果这些逻辑直接压在 Worker 上，在连接数上来或者节点重启集中重连时，容易放大网络处理延迟。所以我单独拆了 Business 线程池，让 IO 尽量保持轻量、业务处理异步下沉。
- 线程数是怎么决定的？
A：我这里不是把线程数当成一个拍脑袋定出来的最优值，而是先按模块职责和 Netty 的常见工程经验给了一个初始配置，再通过压测去验证它在当前环境下是否够用。
具体来说，Boss 线程只负责监听和接收新连接，不参与后续业务处理，所以先配成 1；Worker 线程负责连接上的网络 IO、编解码和 Pipeline 里的轻量处理，这部分本质上是 NIO 事件循环，所以我先按 CPU 核数 来配；Business 线程负责真正的业务逻辑处理，比如握手后的业务处理、在线路由维护、断连清理以及 Redis 交互，这部分不能直接压在 Worker 上，否则业务逻辑一旦变重，就会反过来影响网络读写，所以我单独拆出来，并先给了 CPU * 2 的线程数作为起点。
这里的 CPU 和 CPU * 2 不是说我已经证明它是全局最优，而是一个比较常见、合理的初始值。后面我再结合 RTC 专项压测去验证，比如观察 1000 长连接、高频心跳和节点重启恢复场景下的连接稳定性、错误数、GC 和恢复时间。至少从目前测试结果看，这套线程配置在当前 2C8G 单节点环境下没有暴露出明显瓶颈，所以我认为它对当前规模是够用的。

- 为什么把 JWT 鉴权放在握手阶段，而不是连接建立以后再做业务鉴权？
A：我把 JWT 鉴权放在握手阶段，核心是为了尽早失败，避免非法连接先完成 WebSocket 升级再进入后续处理流程。因为我的 RTC 模块不只是收发消息，后面还要做用户身份绑定、在线路由注册、心跳保活和断连清理，这些逻辑都要求连接一开始就是一个合法用户连接。所以我选择在握手阶段就完成 token 校验和用户身份确认，把未认证请求拦在接入层外面。

- 为什么在线路由要做“本地会话映射 + Redis 全局路由”，而不是只保存本地 Channel？
A：因为只保存本地 Channel 只能解决“当前节点怎么给本机用户发消息”，解决不了“集群里某个用户到底连在哪个节点”。如果没有全局在线路由，跨节点推送时就只能让每个 RTC 节点都尝试投送一遍，代价太大。所以我做成两层：本地会话映射负责节点内快速找到 Channel，Redis 全局路由负责在集群范围内定位用户所在节点，这样才能做到跨节点的精准投送。

- 本地会话表中存放的是什么？
A：本地会话表里存的是 Netty 的 Channel 对象，本质上就是服务器侧对一条客户端连接的抽象。它不是我自己定义的简单实体，而是 Netty 提供的连接对象，后续可以用它判断连接状态、绑定用户属性、执行 writeAndFlush 推送消息，或者在单设备踢人时主动关闭连接。所以本地会话表维护的是 userId -> Channel，用来支持当前节点内的快速定位和直接推送。

- Channel 里为什么还要再绑定 userId 属性？明明本地 Map 已经有 userId -> Channel 了。
A：因为本地会话表解决的是“通过 userId 找 Channel”，但很多运行时场景拿到的起点其实是 Channel 本身，比如断连回调、心跳处理、消息处理。这时候如果提前在 Channel 上绑定了 userId，就可以直接从连接反查出用户身份，方便做断连清理、路由删除和业务处理；否则就只能遍历本地会话表去反查，代价更高。所以我这里实际上做的是一组双向索引。

- 详细说一下时间和次数怎么设置的。
A：我这里把“保活频率”和“死亡判定”分开设计了。客户端 30 秒发一次心跳，是为了维持连接活跃和刷新路由 TTL；服务端并不是丢一次心跳就断开，而是采用 5 分钟读空闲、连续 3 次才关闭的保守策略，也就是大约 15 分钟才会判定连接失活。这样做的好处是能减少网络抖动或短暂异常导致的误踢。与此同时，我又把 Redis 路由 TTL 设为 20 分钟，让正常断连优先走主动清理，异常场景再由 TTL 兜底。

- 那这个 15 分钟是不是太长了
A：是的，如果目标是更实时地反映在线状态，这个值偏保守。当前配置更偏向稳定性优先，适合项目阶段先保证连接不被轻易误踢。如果后续要往生产化优化，我会把空闲剔除时间收紧到心跳周期的 3 到 6 倍，比如 90 到 180 秒，并继续观察误踢率和重连成本。

- 那为什么你项目里最后还是接受了这个偏保守的 15 分钟配置？
A：因为我当前阶段更看重连接稳定性，误踢的代价比晚一点判掉线更高，所以我采用了一个偏保守的空闲关闭配置。客户端 30 秒发一次心跳，服务端不是丢一两次就断，而是 5 分钟读空闲、连续 3 次才关闭，大约 15 分钟判定失活。这样做可以减少网络抖动导致的误踢。同时我又配合了主动断连清理和 Redis TTL 兜底，所以这个值在当前项目阶段是可接受的。后续如果往生产化推进，我会把它进一步收紧到 90 到 180 秒，再结合误踢率和重连成本继续调优。

- 为什么 Redis 全局在线路由要带 TTL，而且还要在断连时主动清理？
A：因为 TTL 和主动清理解决的不是同一个问题，所以两者都要有。主动清理负责正常场景下的及时收敛，TTL 负责异常场景下的兜底自愈。
具体来说，用户正常断连时，服务端是能感知到 channelInactive 的，这时候就应该立刻删除本地会话和 Redis 全局路由。这样做的好处是在线状态收敛快，后续消息不会继续投到已经断开的节点上。如果只靠 TTL，不主动删路由，那即使用户已经离线，Redis 里的路由也还会继续保留一段时间，这段时间内全局在线状态就是滞后的。
但如果只做主动清理、不加 TTL，也有问题。因为在节点 crash、进程被强杀、机器断电这类异常场景下，服务端可能根本来不及走断连回调，Redis 路由就会永久残留，形成脏数据。这样后续系统还会继续把消息路由到一个已经失效的节点。所以 TTL 的作用不是表达精确在线时间，而是保证异常情况下路由最终一定会自己过期，不会无限脏下去。
所以我这里的设计是：主动清理为主，TTL 兜底为辅。 正常情况下优先走断连删除；异常情况下如果来不及删，就靠 TTL 自愈。这样既保证在线状态尽量实时，又避免脏路由永久残留。

- 为什么 RTC 节点重启后会出现重连风暴，你是怎么优化的？
A：因为节点重启时，原来挂在这个节点上的长连接会在很短时间内一起断开。如果客户端都按固定延迟立即重连，就会出现大量连接在同一时刻重新握手，形成瞬时洪峰，这就是重连风暴。它的问题不在于“最终能不能连回来”，而在于恢复窗口里会产生很多无效重试，放大握手、鉴权和网络抖动成本。
我在专项压测里就观察到了这个问题：优化前 1000 连接在节点重启后为了恢复满连，一共发起了 3000 次重连尝试，其中 2000 次失败；后面我把重连策略改成了指数退避 + 随机抖动，让重连请求错峰发起，避免所有连接同一时刻撞回来。优化后同样是 1000 连接重启恢复场景，总尝试次数降到 1126，失败次数降到 126，并且仍然能在约 8s 内恢复到 100% 活跃连接。
更准确地说，这个优化我目前是先在压测脚本这个“模拟客户端”里验证的；如果后续做真实客户端，重连退避、抖动和错误分类逻辑就应该下沉到客户端连接管理层。

- 为什么单节点只能做到快恢复，做不到真正无感？
A：因为单节点重启时，在那段时间里服务本身就是不可用的，客户端没有别的节点可连，所以连接一定会中断。也就是说，单节点能做的是尽快恢复，但做不到真正意义上的“无感”。
如果想接近无感，至少要两台 RTC 节点：一台节点下线或重启时，客户端还能被重新调度到另一台可用节点；再配合消息补拉机制，比如按 seq 或 lastAckSeq 补同步缺失消息，这样连接层虽然短暂中断，但业务层体验可以基本无感。
所以我现在这套方案更准确的定位是：单节点快恢复方案，重点在于降低恢复窗口内的无效重试和抖动；如果要进一步升级到接近无感，就需要多节点接入、节点摘流和重连后消息补偿。
