构建 Netty WebSocket 接入层，采用 Boss / Worker / Business 三层线程模型，实现单设备在线与跨节点精准推送；节点重启后通过指数退避 + 随机抖动重连，将恢复阶段失败重试从 2000 次降至 126 次。
在 2C8G 单节点环境下，RTC 1000 长连接稳定在线，高频心跳场景无异常断连，节点重启后约 8s 恢复至 100% 活跃连接。

一、Netty 基础八股（热身题，不能丢分）

1.1 Reactor 线程模型

**单 Reactor 单线程 / 单 Reactor 多线程 / 主从 Reactor 多线程分别怎么画？你的项目用的哪种？**
（答：
- 单 Reactor 单线程：一个线程同时做 accept、read、decode、compute、encode、send。适合业务逻辑很快的场景（如 Redis）
- 单 Reactor 多线程：一个 Reactor 线程负责 accept + I/O，业务逻辑交给线程池。问题是单 Reactor 线程既要 accept 又要处理 I/O，高并发下成为瓶颈
- 主从 Reactor 多线程：Main Reactor 只负责 accept，Sub Reactor 负责 I/O 读写，业务逻辑交给独立线程池。Netty 的标准模型
- 你的项目：主从 Reactor + 独立 Business 线程组，属于主从 Reactor 的三层变体）

**你代码里是 Boss(1) + Worker(CPU核数) + Business(CPU核数×2)，这属于主从 Reactor 的哪个变体？为什么 Business 要独立出来？**
（答：属于主从 Reactor 的"三级线程模型"。Boss = Main Reactor，Worker = Sub Reactor，Business = 独立业务线程池。Business 独立出来是因为你的业务逻辑有 Redis 调用（路由续期、Lua 脚本）和 Feign HTTP 调用（ACK 回写），这些都是阻塞式网络 I/O。如果在 Worker 线程中执行，会阻塞 NioEventLoop 的 I/O 循环，导致该 Worker 管理的所有 Channel 的读写都被阻塞）

**追问：如果不拆 Business 线程组，Worker 直接处理业务逻辑会怎样？**
（答：Worker 线程被阻塞 → 该 Worker 上注册的所有 Channel 的 I/O 读写全部被阻塞 → 心跳检测失效 → 延迟飙升 → 大量连接超时断开。一个 Worker 管理数千个 Channel，一个 Worker 阻塞影响数千用户）

1.2 NIO 底层

**Selector / Channel / Buffer 三件套各自的角色？Netty 的 NioEventLoop 封装了什么？**
（答：
- Selector：I/O 多路复用器，一个线程监听多个 Channel 的事件（ACCEPT/READ/WRITE）
- Channel：双向通信通道，对应一个网络连接（SocketChannel）或一个监听端口（ServerSocketChannel）
- Buffer：数据缓冲区，NIO 的 ByteBuffer 是读写共用一个指针
- NioEventLoop 封装了：一个 Selector + 一个 Thread + 一个 TaskQueue。循环执行三件事：select → processSelectedKeys → runAllTasks）

**NioEventLoop 的 run() 循环做了哪三件事？**
（答：1) select()：检查是否有就绪的 I/O 事件；2) processSelectedKeys()：处理就绪的 Channel 事件（read/write/accept）；3) runAllTasks()：执行 TaskQueue 中的异步任务（如 ctx.channel().eventLoop().execute(...)提交的任务））

**epoll 空轮询 bug 是什么？Netty 怎么解决的？**
（答：JDK NIO 在 Linux 上使用 epoll 时，即使没有就绪事件，select() 也会立即返回 0，导致 CPU 空转 100%。这是 JDK 的 bug（至今未完全修复）。Netty 的解决方案：记录空轮询次数，当连续空轮询超过阈值（默认 512 次）后，重建 Selector——把旧 Selector 上注册的所有 Channel 迁移到新 Selector 上）

**Linux 下 Netty 默认用 epoll 还是 poll？你的项目用的 NioEventLoopGroup，如果部署在 Linux 上为什么不用 EpollEventLoopGroup？性能差多少？**
（答：NioEventLoopGroup 在 Linux 上底层用的是 epoll（通过 JDK 的 Selector 实现）。EpollEventLoopGroup 是 Netty 直接调用 Linux epoll 系统调用，绕过 JDK 的 Selector 抽象层。性能差异约 5~10%：EpollEventLoopGroup 支持 Edge-Triggered 模式、SO_REUSEPORT 等高级特性。你的项目没用 EpollEventLoopGroup 是因为 NioEventLoopGroup 跨平台兼容，且 1000 连接的场景下性能差异不明显）

1.3 ByteBuf 与零拷贝

**ByteBuf 和 JDK ByteBuffer 的区别？**
（答：1) 双指针：ByteBuf 有 readerIndex 和 writerIndex，读写独立不需要 flip()。ByteBuffer 只有一个 position 指针，读写需要手动切换；2) 池化：ByteBuf 支持 PooledByteBufAllocator，重复利用已分配的内存，减少 GC 压力；3) 引用计数：ByteBuf 通过 ReferenceCounted 接口做引用计数管理，手动 release() 释放；4) 动态扩容：ByteBuf 写满后自动扩容，ByteBuffer 容量固定；5) 支持 Direct 和 Heap 两种内存模式）

**Netty 的零拷贝体现在哪几个层面？**
（答：1) CompositeByteBuf：合并多个 ByteBuf 时不拷贝底层数据，只创建虚拟视图；2) FileRegion：文件传输时直接从内核缓冲区发送到 Socket 缓冲区，不经过用户态拷贝（底层用 sendfile 系统调用）；3) wrap/slice：包装已有字节数组或截取 ByteBuf 片段时不拷贝数据，共享底层存储）

**你的项目用的是 TextWebSocketFrame，底层 ByteBuf 的引用计数谁来释放？SimpleChannelInboundHandler 和 ChannelInboundHandlerAdapter 在释放上有什么区别？**
（答：你的 MessageInboundHandler 继承 SimpleChannelInboundHandler，它在 channelRead() 方法中自动调用 ReferenceCountUtil.release(msg) 释放 ByteBuf。如果用 ChannelInboundHandlerAdapter，需要手动释放，否则内存泄漏。WebSocketTokenAuthHandler 继承 ChannelInboundHandlerAdapter，调用 super.channelRead(ctx, msg) 传给下游，不在这一层释放）

1.4 内存模型与池化

**PooledByteBufAllocator vs UnpooledByteBufAllocator 的区别？Netty 4.x 默认用哪个？**
（答：PooledByteBufAllocator 预分配内存块，通过对象池复用，减少频繁 malloc/free 和 GC。UnpooledByteBufAllocator 每次分配新内存。Netty 4.1+ 默认用 PooledByteBufAllocator。池化适合高频小数据传输（如 IM 消息），非池化适合低频大数据传输）

**jemalloc 的 Arena / Chunk / Page / Subpage 分配策略概述**
（答：Netty 的 PooledByteBufAllocator 借鉴了 jemalloc 的设计。Arena：每个线程绑定一个 Arena，减少锁竞争。Chunk：Arena 内以 Chunk（默认 16MB）为单位向操作系统申请内存。Page：Chunk 被切分为 Page（默认 8KB）。Subpage：小于 Page 的请求（如 256 字节），Page 进一步切分为 Subpage。分配策略：>=8KB 走 Page 级别分配，<8KB 走 Subpage 级别分配）

**什么场景下会出现内存泄漏？怎么检测？**
（答：场景：Handler 中读取了 ByteBuf 但忘记 release()，或者 Pipeline 中间某个 Handler 拦截了消息但没有传给下游也没有释放。检测：启动时加 JVM 参数 -Dio.netty.leakDetection.level=PARANOID，Netty 会对每个 ByteBuf 做引用追踪，泄漏时打印详细堆栈。四个级别：DISABLED < SIMPLE < ADVANCED < PARANOID，生产环境建议 SIMPLE，开发环境用 PARANOID）

二、Channel Pipeline 设计（你的核心架构）

2.1 Pipeline 中每个 Handler 的职责

**你的 Pipeline 顺序（NettyServer.java:78-85）：**

| 顺序 | Handler | 所在线程组 | 职责 |
|------|---------|-----------|------|
| 1 | IdleStateHandler(300, 0, 0) | Worker | 读空闲检测，5分钟无数据触发事件 |
| 2 | HttpServerCodec | Worker | HTTP 编解码（WebSocket 握手前是 HTTP） |
| 3 | ChunkedWriteHandler | Worker | 大数据分块传输支持 |
| 4 | HttpObjectAggregator(65536) | Worker | 聚合 HTTP 请求，最大 64KB |
| 5 | WebSocketTokenAuthHandler | Worker | 自定义鉴权，拦截 HTTP Upgrade 请求 |
| 6 | WebSocketServerProtocolHandler | Worker→Business | WebSocket 握手/帧编解码 |
| 7 | MessageInboundHandler | Business | 业务逻辑处理 |

可能被问到的问题：

**为什么 IdleStateHandler 放在最前面？如果放在 HttpServerCodec 之后呢？**
（答：放在最前面可以在字节级别检测空闲，不依赖编解码是否成功。放在编解码后面也可以，但如果编解码异常阻塞了，idle 检测就失效了）

**HttpObjectAggregator 的 65536 怎么定的？这个值限制的是什么？**
（答：限制 HTTP 请求体的最大聚合长度。对 WebSocket 来说只影响握手阶段的 HTTP Upgrade 请求，64KB 足够了。WebSocket 帧不受此限制）

**WebSocket 握手完成后，HttpServerCodec 和 HttpObjectAggregator 还在 Pipeline 里吗？**
（答：WebSocketServerProtocolHandler 握手成功后会自动移除 HTTP 相关 Handler——HttpServerCodec、HttpObjectAggregator、ChunkedWriteHandler 都会被移除，因为握手后走 WebSocket 帧协议，不再需要 HTTP 编解码）

**WebSocketTokenAuthHandler 为什么放在 WebSocketServerProtocolHandler 前面？**
（答：必须在 WebSocket 握手之前拦截 HTTP Upgrade 请求，提取 query 中的 token 进行鉴权。如果放在后面，请求已经被 WebSocket 协议处理器拦截了，鉴权 Handler 收不到 FullHttpRequest）

**@ChannelHandler.Sharable 是什么意思？你的 MessageInboundHandler 和 WebSocketTokenAuthHandler 都标了这个注解，它们为什么能被多个 Channel 共享？有没有线程安全问题？**
（答：@Sharable 表示一个 Handler 实例可以被添加到多个 Channel 的 Pipeline 中。无状态或线程安全的 Handler 才能标 Sharable。你的两个 Handler 没有实例级别的可变状态——userId 存在 Channel Attribute 中（每个 Channel 独立），idleCounter 也存在 Channel Attribute 中。Redis 和 Feign 调用本身是线程安全的。所以可以 Sharable）

2.2 Handler 线程归属

**pipeline.addLast(businessGroup, messageInboundHandler) 这行代码的含义？**
（答：把 messageInboundHandler 绑定到 businessGroup 线程组执行。这意味着 messageInboundHandler 的所有回调方法（channelRead0、userEventTriggered、channelInactive 等）都在 businessGroup 的线程中执行，而不是在 Worker 的 NioEventLoop 中执行。实现了 I/O 线程和业务线程的分离）

**Worker 线程和 Business 线程之间的切换成本是什么？**
（答：涉及一次 task queue 投递。Worker 线程把消息封装成 Runnable 投递到 Business EventExecutor 的 taskQueue 中，Business 线程从队列取出执行。有一次上下文切换的开销，但远小于业务逻辑阻塞 Worker 线程的代价）

**如果 businessGroup 的所有线程都在忙，新消息会怎样？**
（答：堆积在对应 EventExecutor 的 taskQueue 里（无界队列），不会阻塞 Worker 的 I/O 线程。Worker 照常做 select 和 processSelectedKeys，只是消息的业务处理会延迟。极端情况下 taskQueue 堆积可能导致内存溢出，但 4 个 Business 线程在 1000 连接场景下远远够用）

三、WebSocket 鉴权方案深挖

3.1 你的鉴权流程（WebSocketTokenAuthHandler.java）

```
客户端 → ws://host:9101/api/v1/chat/message?token=xxx
         ↓
1. ChannelInboundHandlerAdapter.channelRead() 拦截 FullHttpRequest
2. 从 URI query 中提取 token（UrlQuery.of(uri).get("token")）
3. JwtUtil.parse(token) → Claims
4. 从 Claims 取 userId（claims.getSubject()）
5. 检查 Redis jwt:logout:{userId}（判断 token 的 iat 是否在登出时间之后）
6. 通过 → userId 写入 Channel Attribute，URI 去掉 query 参数后传给下游
7. 失败 → 返回 401 HTTP 响应 + close
```

可能被问到的问题：

**为什么不用 HTTP Header（Authorization: Bearer xxx）传 token？**
（答：浏览器 WebSocket API（new WebSocket(url)）不支持自定义 Header，只能走 URL query 或 Sec-WebSocket-Protocol 子协议。URL query 是最通用的方式。生产环境建议用 wss（TLS 加密）防止 token 在网络中被截获，同时要注意 token 可能出现在服务器 access log 和代理日志中）

**JWT 签名用的 HS512，密钥硬编码在 ConfigEnum 枚举里，面试官会追问：生产环境怎么管理密钥？**
（答：应该走配置中心（Nacos Config）或 KMS（Key Management Service），不应硬编码在代码中。密钥硬编码意味着：1) 代码泄露 = 密钥泄露；2) 密钥轮换需要改代码重新部署。这是你的项目待优化点，面试时主动承认并说出改进方案）

**jwt:logout:{userId} 这个 Redis key 的作用？token 还没过期但用户已经登出了怎么办？**
（答：这就是 JWT 无状态的痛点——JWT 一旦签发，在过期前无法主动撤销。你的方案是在 Redis 存登出时间戳（jwt:logout:{userId} = 登出时间），鉴权时判断 token 的签发时间 iat 是否早于登出时间。如果 iat < logoutAt，说明 token 是在登出之前签发的，判定为失效。这本质上是 JWT 黑名单方案的变体——只存一个时间戳而不是存所有失效 token）

**JWT 过期时间 3 小时，WebSocket 长连接如果超过 3 小时，连接还在但 token 过期了怎么办？**
（答：当前实现中连接建立后不再校验 token 过期。鉴权只在 WebSocket 握手阶段做一次。连接建立后依赖路由 TTL 20 分钟续期 + 心跳机制维持。如果需要更严格的安全策略，可以在心跳中重新校验 token 有效性，或者定期让客户端刷新 token）

**鉴权这一步有 Redis 调用，如果 Redis 挂了怎么办？当前代码会怎样？**
（答：Redis 调用抛异常 → catch 块 → 返回 401 关闭连接。结果是所有用户都连不上。容灾策略：1) 如果 Redis 不可用，降级为不检查登出状态，只校验 JWT 签名和过期时间；2) 使用 Redis Sentinel/Cluster 做高可用）

四、三层线程模型详解

4.1 参数设置与依据

| 线程组 | 线程数 | 你的代码 | 设置依据 |
|--------|--------|---------|---------|
| Boss | 1 | new NioEventLoopGroup(1) | 只监听一个端口，1个线程足够 |
| Worker | CPU 核数 | NioEventLoopGroup(NettyRuntime.availableProcessors()) | I/O 密集型，一般 = CPU 核数 |
| Business | CPU 核数 × 2 | DefaultEventExecutorGroup(NettyRuntime.availableProcessors() * 2) | 业务逻辑有 Redis / RocketMQ 调用，涉及网络 I/O 等待，需要更多线程 |

可能被问到的问题：

**你的服务器是 2C8G，那 Worker = 2, Business = 4，总共才 7 个线程，能撑多少连接？**
（答：线程数不等于并发连接数。NIO 的核心优势就是一个线程通过 Selector 可以管理数万个 Channel。实测 2 个 Worker 线程管理 1000 个长连接完全没有压力。瓶颈不在线程数而在内存和 CPU）

**Worker 和 Business 的比例为什么是 1:2？有没有做过调优？如果 Business 线程全被 Feign 调用阻塞了，Worker 还能正常工作吗？**
（答：Worker 不受影响——Worker 只做 I/O 读写和编解码，不执行 Business 的任务。Business 阻塞只会导致 MessageInboundHandler 的处理堆积。1:2 是经验值——Business 有网络等待（Redis/Feign），需要比 Worker 更多的线程来掩盖等待时间。实际应该根据压测数据调整）

**NettyRuntime.availableProcessors() 在容器（Docker）里返回的是什么？会不会返回宿主机的核数？**
（答：JDK 8u191+ / JDK 10+ 会正确识别容器 CPU 限制（cgroups）。如果低版本 JDK 8，会返回宿主机的核数，导致线程数过多。需要手动指定 -XX:ActiveProcessorCount=2 或在代码中硬编码）

**DefaultEventExecutorGroup 和 NioEventLoopGroup 的区别？为什么 Business 不用 NioEventLoopGroup？**
（答：NioEventLoopGroup 每个线程都绑定一个 Selector，用于 I/O 多路复用。DefaultEventExecutorGroup 每个线程只有一个 TaskQueue，是纯任务执行线程池。Business 不需要 Selector，只需要一个线程池来执行业务逻辑，所以用 DefaultEventExecutorGroup 更轻量）

4.2 ServerBootstrap 参数

**SO_BACKLOG = 128：TCP 全连接队列大小**
（答：TCP 三次握手完成后，连接放入全连接队列等待 accept()。128 是 Linux 默认值。半连接队列（SYN Queue）存放收到 SYN 但未完成三次握手的连接，大小由 net.ipv4.tcp_max_syn_backlog 控制。SYN Flood 攻击通过发大量 SYN 不完成握手来占满半连接队列，防御方式：开启 SYN Cookie（net.ipv4.tcp_syncookies=1））

**追问：SO_BACKLOG 设 128，你有 1000 个用户同时重连（比如节点重启恢复场景），会不会有部分连接被拒绝？**
（答：可能。1000 个重连请求瞬间到来时，如果 accept() 处理速度跟不上，超过 128 的连接会被内核丢弃（发 RST 或静默丢弃取决于配置）。你的指数退避 + 随机抖动策略把重连请求打散在时间轴上，降低了瞬时并发，实测 128 够用。生产环境可以调大到 1024）

**SO_KEEPALIVE = true：TCP 层 keep-alive**
（答：TCP keep-alive 默认探测间隔 2 小时（tcp_keepalive_time=7200s），远大于你的应用层心跳。TCP keep-alive 的作用是检测对端进程崩溃或网络不可达——如果对端 OS 还在但进程挂了，TCP 层能通过 keep-alive 发现。应用层心跳检测的是"对端应用是否正常响应"。两者互补，不冲突）

**TCP_NODELAY = true：禁用 Nagle 算法**
（答：Nagle 算法会攒小包合并发送——发送方在未收到前一个包的 ACK 之前，不会发送下一个小包，而是等攒够 MSS 大小或收到 ACK 后一起发。好处是减少小包数量节省带宽，坏处是增加延迟。IM 场景每条消息很小（几百字节）但要求低延迟，必须关闭 Nagle）

**有没有设置 WRITE_BUFFER_WATER_MARK？如果推送消息速度大于网络发送速度，Channel 的写缓冲区会无限增长吗？**
（答：当前代码没有设置水位线。如果推送速度 > 网络发送速度，ChannelOutboundBuffer 会持续增长，最终 OOM。生产环境应该设置 ChannelOption.WRITE_BUFFER_WATER_MARK，默认高水位 64KB / 低水位 32KB。当写缓冲区超过高水位，Channel.isWritable() 返回 false，应该暂停推送；低于低水位后恢复。需要在代码中监听 channelWritabilityChanged 事件）

五、心跳与空闲检测机制

5.1 你的心跳方案（MessageInboundHandler.java:96-140）

```
Server 端 IdleStateHandler(readerIdleTime=300s, writerIdleTime=0, allIdleTime=0)
    ↓ 300秒无读事件
触发 IdleStateEvent(READER_IDLE)
    ↓
userEventTriggered() 处理：
    idleCounter++
    if idleCounter < 3:
        发送 SERVER_PING → 客户端
    else:
        强制关闭连接
```

可能被问到的问题：

**为什么是服务端主动 PING，而不是客户端主动 PING？**
（答：实际上是双向的。客户端也会发 HEART_BEAT(type=5)，默认 30 秒一次。服务端的 IdleStateHandler 是"兜底检测"——如果客户端 300 秒内没发任何数据（包括心跳和正常消息），服务端才主动发 SERVER_PING 探活。正常情况下客户端 30 秒心跳间隔，服务端 300 秒空闲检测永远不会触发）

**300 秒（5 分钟）的读超时怎么定的？太长会不会导致死连接检测太慢？**
（答：IM 场景通常 30s~120s。5 分钟对于 NAT 穿透来说可能太长了——很多运营商 NAT 设备 4 分钟就会清理映射。但你的客户端每 30 秒发心跳，30 秒就会有一次读事件，所以 300 秒的 idle 检测实际上是在客户端心跳完全停止的场景下才触发。简历里写的 1000 连接稳定无异常断连，是因为测试环境没有 NAT——直连 RTC 节点，不经过运营商 NAT）

**MAX_IDLE_COUNT = 3，所以最长 15 分钟才能检测到死连接？这对 IM 场景可接受吗？**
（答：最坏情况：客户端刚好在第一次 idle 检测之前发了一条消息 → 重置计时 → 再等 300s → idle 1 → 300s → idle 2 → 300s → idle 3 → 关闭。总计 900s + α ≈ 15 分钟。对生产环境偏长。可以调短 readerIdleTime 到 60~120s，MAX_IDLE_COUNT 保持 3，总检测时间缩短到 3~6 分钟）

**Idle 计数器存在 Channel Attribute 里，有线程安全问题吗？**
（答：没有。关键原因：同一个 Channel 绑定到同一个 EventExecutor。pipeline.addLast(businessGroup, handler) 会把 Channel 绑定到 businessGroup 中的某一个固定的 EventExecutor。该 Channel 的所有事件（IdleStateEvent、channelRead0）都由同一个线程处理。所以 attr.get() 和 attr.set() 不会并发，无需加锁）

**WebSocket 协议本身有 Ping/Pong 帧，你为什么不用 WebSocket 原生的 Ping/Pong？**
（答：WebSocket Ping/Pong 是协议层的，WebSocketServerProtocolHandler 默认会自动回 Pong。但应用层心跳可以携带自定义数据（比如最后消息 seq、在线状态等），更灵活地控制超时策略。两者可以共存——WebSocket Ping/Pong 做协议级保活，应用层心跳做业务级探活）

六、会话管理与分布式路由

6.1 本地会话管理（NettySessionManager.java）

数据结构：ConcurrentHashMap<Long, Channel> — userId → Channel 映射

**为什么用 ConcurrentHashMap 而不是普通 HashMap？哪些线程会并发访问这个 Map？**
（答：三种线程并发访问：1) Worker 线程触发 channelInactive → remove(channel)；2) Business 线程触发 HandshakeComplete → add(userId, channel)；3) HTTP PushController 线程（Tomcat）→ getChannel(userId)。多线程并发读写，必须用线程安全容器。ConcurrentHashMap 的分段锁机制在读多写少场景下性能最好）

**remove(Channel channel) 方法里用了 userChannelMap.remove(userId, channel)，为什么用双参数 remove？**
（答：CAS 语义——只有当前 Map 中 userId 对应的 value 确实是这个 channel 时才删除。场景：用户 A 在节点上用旧连接 C1，然后用新连接 C2 登录。C2 注册后 C1 的 channelInactive 触发。如果用单参数 remove(userId)，会把 C2 的映射也删掉。双参数 remove(userId, C1) 保证只删旧连接，不影响新连接。这是一个非常好的设计点，面试时主动讲出来加分）

6.2 分布式路由注册

Redis Key: `im:route:{userId}` → nodeId，TTL = 1200 秒（20分钟）

路由注册用的 Lua 脚本（MessageInboundHandler.java:42-45）：
```lua
local oldNode = redis.call('GET', KEYS[1])
redis.call('SET', KEYS[1], ARGV[1], 'EX', tonumber(ARGV[2]))
return oldNode
```

**为什么要用 Lua 脚本而不是先 GET 再 SET？**
（答：原子性。如果分成两个命令：GET 获取 oldNode，SET 写入 newNode——两步之间可能另一个节点也在处理同一用户的连接，插入了自己的 SET，你的 SET 又覆盖了它。Lua 脚本在 Redis 中原子执行，GET 和 SET 之间不会被其他命令打断）

**路由 TTL 20 分钟怎么定的？和心跳超时 5 分钟 × 3 次 = 15 分钟的关系？**
（答：20 分钟 > 15 分钟，确保连接真正断开（idle 检测完成）后路由才过期。同时每次收到客户端消息都会续期 TTL（MessageInboundHandler.java:67 的 redisTemplate.expire），所以活跃连接的路由永远不会过期。只有客户端完全无响应 15 分钟 → 连接被关闭 → 最多再等 5 分钟路由过期）

**路由过期后但用户连接还在怎么办？**
（答：正常情况下不会发生——每次客户端发消息都续期 TTL。只有客户端 15 分钟完全无操作才可能。此时服务端 idle 检测会先关闭连接（3 次 × 5 分钟 = 15 分钟），触发 channelInactive → 清理路由。路由 TTL 20 分钟是额外的安全兜底）

**如果 Redis 宕机，路由全丢了，服务怎么恢复？**
（答：当前没有容灾机制。Redis 宕机后：1) 新连接无法注册路由；2) 消息发送时查路由失败，所有消息都会走 OFFLINE 通道。用户重连后重新注册路由即可恢复。可以考虑 Redis Sentinel 或 Cluster 做高可用）

6.3 负载上报

Redis 结构：Sorted Set `im:rtc:load`，member = wsUrl, score = 在线用户数

```java
// 每次 add/remove 用户后调用（NettySessionManager.java:106-111）
redisTemplate.opsForZSet().add("im:rtc:load", getLocalWsUrl(), userChannelMap.size());
```

**Gateway 怎么用这个 ZSet 做负载均衡？**
（答：Gateway 登录接口返回 nettyUrl 给客户端时，从 ZSet 中取 score 最小的节点（在线用户最少），实现最小连接数负载均衡。命令：ZRANGEBYSCORE key -inf +inf LIMIT 0 1，或 ZRANGEWITHSCORES key 0 0 取 score 最小的 member）

**每次 add/remove 都写 Redis 一次，1000 QPS 下会不会成为瓶颈？**
（答：会有性能影响但不是瓶颈——ZADD 是 O(log N) 操作，单次约 0.1ms。1000 连接的 add/remove 频率远低于 1000 QPS（大部分时间连接是稳定的）。但如果要优化，可以做本地聚合 + 定期上报（比如每 5 秒上报一次当前在线数），减少 Redis 写入频率）

**节点宕机后 ZSet 里的 member 怎么清理？@PreDestroy 只在正常停机时触发，kill -9 呢？**
（答：kill -9 不会触发 @PreDestroy，ZSet 中该节点的 member 会残留。需要 TTL 机制或定时巡检脚本。改进方案：1) 给 ZSet member 加 TTL（需要额外 key）；2) Gateway 选节点时先 ping 一下健康检查接口；3) 定时任务扫描 ZSet 中的节点，检查 Nacos 注册状态，清理已下线节点）

七、单设备在线（挤占登录）

7.1 你的方案（MessageInboundHandler.java:116-138）

```
HandshakeComplete 事件触发：
1. Lua 原子 swap：im:route:{userId} = 当前nodeId，返回 oldNodeId
2. 取出本地旧 Channel（sessionManager.getChannel）
3. 注册新 Channel 到 sessionManager
4. 分支判断：
   a) oldNodeId 存在 且 != 本节点 且 != "OFFLINE"
      → Redis pub/sub 通知旧节点踢人（"userLogout" channel, message = oldNodeId:userId）
   b) oldChannel 存在 且 != 新Channel
      → 直接 oldChannel.close()
```

**为什么先获取旧 Channel 再注册新 Channel，而不是反过来？**
（答：防止在注册新 Channel 后，旧 Channel 的 channelInactive 触发 remove(channel) 把新 Channel 的映射也删了。但因为用了 remove(userId, channel) 的 CAS 语义，旧 Channel 的 remove 只会删自己不会删新的，所以先后顺序影响不大。代码这样写语义更清晰——先拿到旧状态再更新）

**跨节点踢人走 Redis Pub/Sub 有什么问题？**
（答：Pub/Sub 消息不持久化。如果目标节点刚好断开了 Redis 订阅（比如网络闪断），踢人消息就丢了。结果是旧节点上的连接不会被关闭，出现 20 分钟内的双设备在线。路由 TTL 过期后旧路由失效，消息不再推给旧连接，但连接本身还在直到 idle 超时。对 IM 场景来说大部分可接受，严格场景可用 Redis Stream 替代 Pub/Sub）

**Redis Pub/Sub 所有节点都会收到消息，你怎么过滤？有没有优化空间？**
（答：消息格式是 `NODE_2:12345`，UserKickoutMessageListener 收到后解析 targetNodeId，判断是否等于 localNodeId。非目标节点做一次字符串解析后忽略。优化方案：1) 频道按节点分：userLogout:NODE_1，每个节点只订阅自己的频道；2) 用 Redis Stream + Consumer Group 替代 Pub/Sub，精确投递）

八、消息推送链路（RocketMQ 集成）

8.1 Topic 设计

| Topic | 用途 | 消费模式 | Tag |
|-------|------|---------|-----|
| IM_CHAT | 聊天消息 | CONCURRENTLY | nodeId（定向投递到目标节点） |
| IM_CHAT:OFFLINE | 离线回流 | — | OFFLINE |
| IM_NOTIFY | 通知类（好友申请/朋友圈等） | CONCURRENTLY | nodeId |
| IM_NOTIFY:OFFLINE | 通知离线回流 | — | OFFLINE |

**为什么 IM_CHAT 和 IM_NOTIFY 要分两个 Topic？能不能合一个？**
（答：隔离不同业务的消费速率和重试策略。聊天消息是核心链路不能丢，需要更多消费线程和更严格的重试。通知类容忍度更高——朋友圈提醒丢了用户刷新一下就好。分 Topic 后两者互不影响，聊天消息堆积不会拖慢通知推送）

**selectorExpression = "${im.node.id:NODE_1}" 意味着每个节点只消费 tag=自己 nodeId 的消息。那如果某个节点宕机了，它 tag 的消息谁消费？**
（答：没人消费，堆积在 Broker 直到消息过期（默认 48 小时）。消息发送端已经做了路由查询，如果用户不在线会走 OFFLINE tag。但如果消息发送时用户在线（路由指向宕机节点），发完后节点马上宕机 → 消息堆积 → 用户重连到其他节点后路由更新，但之前堆积的消息不会被消费。需要消费端 ACK + 补偿机制，或者用户重连后走 /sync 接口补齐）

**ConsumeMode 为什么用 CONCURRENTLY 而不是 ORDERLY？**
（答：RTC 的消息消费只需要 writeAndFlush 推给客户端。推送操作很快（微秒级），不需要严格排序。客户端收到消息后按 seq 排序展示。ORDERLY 消费会加锁串行处理，降低吞吐量）

8.2 推送失败处理

**IM_CHAT 为什么用 syncSend 回流离线，而 IM_NOTIFY 用 asyncSend？**
（答：聊天消息更重要——syncSend 等待 Broker 确认，确保消息确实进了离线队列。通知类允许丢失——asyncSend 火速发完不等确认，即使发送失败也不重试）

九、优雅停机（Graceful Shutdown）

9.1 你的停机流程（NettyServer.java:94-127）

```
@PreDestroy
1. sleep(5000ms) — 等待服务从注册中心摘除
2. 遍历所有在线用户，发送 LOG_OUT 消息（"服务器维护中，请稍后重连"）
3. sleep(3000ms) — 等待客户端处理断开消息
4. bossGroup.shutdownGracefully()
5. workerGroup.shutdownGracefully()
6. businessGroup.shutdownGracefully()
7. syncUninterruptibly() 等待完成
```

**5 秒等待够不够？Nacos 的心跳间隔是多少？服务摘除需要多久？**
（答：Nacos 默认心跳 5s，15s 标记不健康，30s 摘除实例。所以 5 秒可能不够——Gateway 可能还在把新请求路由到正在停机的节点。应该等 Nacos 摘除后（约 30s）再断连接。但 30s 的 sleep 会让 Spring 的 @PreDestroy 很慢，需要权衡）

**Thread.sleep 在 @PreDestroy 里合适吗？Spring 的销毁有超时限制吗？**
（答：Spring Boot 默认 @PreDestroy 有 30 秒超时（spring.lifecycle.timeout-per-shutdown-phase=30s）。你的代码 sleep 5s + 3s + shutdownGracefully 默认 15s = 23s，在超时范围内。但如果 shutdownGracefully 执行更久，可能被 Spring 强制中断）

**shutdownGracefully() 的默认超时参数是什么？**
（答：shutdownGracefully(quietPeriod, timeout, unit)。默认 quietPeriod=2s, timeout=15s。quietPeriod 表示"在这段时间内如果没有新任务提交，就认为可以安全关闭"。timeout 是最大等待时间）

**停机期间如果有 RocketMQ 消息正在消费中怎么办？**
（答：Netty 关闭后 channel.writeAndFlush() 会抛 ClosedChannelException。消费线程 catch 后，RocketMQ 认为消费失败 → 重试。当前 max-reconsume-times 默认 16 次，最终进死信队列。但如果用户已经重连到其他节点，重试时路由已更新，消息会被投递到新节点成功消费）

十、指数退避 + 随机抖动重连策略

10.1 你简历中写的指标

> 节点重启后通过指数退避 + 随机抖动重连，将恢复阶段失败重试从 2000 次降至 126 次

**重连逻辑是在客户端还是服务端？**
（答：客户端。服务端重启后客户端检测到断连触发重连。当前是在压测脚本 ws_load_test.js 这个模拟客户端里实现并验证的。如果后续做真实客户端（iOS/Android），重连逻辑应该下沉到客户端连接管理层）

**这个重连代码在你的项目哪里？**
（答：scripts/ws_load_test.js，自写的 Node.js 脚本，基于 ws 库。重连退避逻辑在 buildReconnectDelay 函数中，参数全部可配置）

指数退避的具体参数：

| 参数 | 值 | 含义 |
|------|-----|------|
| reconnectDelayMs | 1000ms | 初始重连延迟 |
| reconnectMaxDelayMs | 10000ms | 最大重连延迟（封顶） |
| reconnectBackoffFactor | 2 | 指数底数 |
| reconnectJitterMs | 1000ms | 随机抖动上限 |
| maxReconnectAttempts | 0（无限） | 不限重试次数 |

退避公式（ws_load_test.js:211-228）：
```
cappedDelay = min(baseDelay × 2^(attempt-1), maxDelay)
finalDelay  = uniform_random(minDelay, cappedDelay + jitter)
```

具体退避轨迹：

| 重试次数 | cappedDelay | 实际范围（加 jitter） |
|---------|-------------|---------------------|
| 第1次 | 1s | 1s ~ 2s |
| 第2次 | 2s | 1s ~ 3s |
| 第3次 | 4s | 1s ~ 5s |
| 第4次 | 8s | 1s ~ 9s |
| 第5次+ | 10s（封顶） | 1s ~ 10s |

**2000 次降到 126 次是怎么来的？**

优化前（固定 1s 延迟，无退避无抖动）实际压测日志：
```
04:03:50 active=0/1000 attempted=1000 errors=0     ← 稳态 1000 连接，执行 kill 停机
04:03:53 active=0/1000 attempted=2000 errors=998    ← 第一波重连全失败（服务还没起来）
04:03:55 active=0/1000 attempted=2998 errors=1910   ← 第二波继续失败
04:03:57 active=0/1000 attempted=3000 errors=2000   ← 3 轮固定重试打完，累计 2000 失败
04:03:59 active=291/1000 attempted=4000 errors=2000 ← 服务恢复，291 先连上
04:04:01 active=1000/1000 attempted=4000 errors=2000 ← 全部恢复，kill → 满连约 11 秒
```

优化后（指数退避 + 随机抖动）：attempted=1126, errors=126, 约 8s 恢复满连。

核心收益不是恢复速度快了 3 秒，而是无效重试从 2000 降到 126——降低了恢复窗口内对服务端的冲击。

**随机抖动（jitter）的目的是什么？**
（答：防止多个客户端的退避时间完全对齐形成"周期性洪峰"。纯指数退避时 1000 个客户端的第 N 次重试仍然会在同一时刻到来（因为大家的退避计算结果一样）。加 jitter 后重连请求被打散在时间轴上，避免服务刚恢复就被瞬时洪峰打垮）

十一、选型对比（面试高频题）

11.1 Netty vs 原生 NIO vs Mina vs gRPC-streaming

| 维度 | Netty | 原生 NIO | Mina | gRPC |
|------|-------|---------|------|------|
| 开发效率 | 高（Pipeline 抽象） | 低（裸写 Selector） | 中 | 中 |
| 性能 | 极高 | 高但易出 bug | 一般 | 高 |
| 社区 | 极活跃 | JDK 自带 | 停滞 | Google 维护 |
| WebSocket 支持 | 原生 | 手写 | 有但陈旧 | 不直接支持 |
| IM 场景适配 | 最佳 | 太底层 | 落后 | 适合微服务间调用 |

11.2 WebSocket vs TCP 长连接 vs SSE vs 长轮询

**为什么选 WebSocket？**
（答：双向通信、浏览器原生支持、头部开销小（2~10 字节 vs HTTP 几百字节）。IM 场景需要服务端主动推送消息给客户端，WebSocket 是最佳选择）

**为什么不用纯 TCP？**
（答：浏览器不支持原生 TCP Socket、需要自己设计应用层协议（帧分隔、粘包拆包）。如果是纯 App 端可以考虑自定义 TCP 协议，性能更高）

**为什么不用 SSE（Server-Sent Events）？**
（答：SSE 是单向的（server → client），客户端发消息还需要额外 HTTP 请求。不适合 IM 的双向实时通信。适合只需要服务端推送的场景如股票行情、日志流）

11.3 Redis Pub/Sub vs RocketMQ 在跨节点通信中的选择

**你的架构中两者都用了，踢人走 Redis Pub/Sub，消息推送走 RocketMQ。为什么不统一？**
（答：踢人是低频、低可靠性要求的广播消息，Pub/Sub 延迟最低（微秒级）。消息推送是高频、高可靠性要求的单播消息，需要 MQ 的持久化、重试和消费确认能力。用 Pub/Sub 做消息推送的问题：消息不持久化，消费者断开期间的消息全丢）

十二、协议设计

12.1 客户端 → 服务端
```json
{
  "type": 1,    // ClientMessageTypeEnum: 1=ACK, 2=LOG_OUT, 5=HEART_BEAT
  "data": {...}  // type-specific payload
}
```

12.2 服务端 → 客户端
```json
{
  "type": 2,    // PushTypeEnum: 1=新会话, 2=新消息, 3=朋友圈, 4=好友申请
  "data": {...}
}
```

**为什么客户端和服务端的 type 编码是两套不同的枚举？不会混淆吗？**
（答：职责清晰。客户端只发控制类消息（ACK、心跳、登出），服务端推送业务消息（新消息、新会话、通知）。两套 type 互不冲突。客户端发的 type=1 是 ACK，服务端推的 type=1 是新会话通知——收发方向不同，不存在混淆）

**消息体为什么用 JSON 而不是 Protobuf？带宽开销大多少？**
（答：JSON 可读性好、调试方便（Wireshark/Chrome DevTools 直接看）。Protobuf 体积小约 30~50%，且有严格的 schema 校验。1000 QPS 级别 JSON 够用——每条消息约 500 字节，1000 QPS = 500KB/s 带宽，在千兆网卡面前可以忽略。如果到万级以上 QPS，可以考虑换 Protobuf 减少带宽和序列化开销）

**有没有消息压缩？WebSocket 支持 Per-Message Compression Extension，你有没有开启？**
（答：当前没有开启 WebSocket 压缩。WebSocketServerProtocolHandler 可以配置 allowExtensions=true 开启 permessage-deflate 压缩。开启后每条消息 zlib 压缩，减少约 60~80% 带宽，但增加 CPU 开销。1000 连接场景下带宽不是瓶颈，没有开启的必要）

**MessageDTO 的 data 字段是 Object 类型，反序列化时怎么知道具体类型？**
（答：通过 type 字段判断——type=1 时 data 是 ACK 数据（sessionId+seq），type=5 时 data 是心跳字符串。客户端和服务端约定好每个 type 对应的 data 结构，按 type 分发解析。JSON 的灵活性让 data 可以是任意结构，不需要预定义所有类型）

十三、ACK 机制

13.1 你的 ACK 方案（MessageInboundHandler.java:142-164）

客户端收到消息后发 ACK:
- 方式1: `{ sessionId, seq }` → Feign 调用 MessageService.ackBySeq()
- 方式2: `{ messageId/msgUuid }` → Feign 调用 MessageService.ackByMessageId()

**ACK 处理走 Feign（MessageAckClient.java），这是同步 HTTP 调用，在 Business 线程里会阻塞吗？**
（答：会阻塞 Business 线程直到 Feign 返回。如果 Messaging 服务响应慢（比如 MySQL 更新慢），Business 线程会被阻塞，导致该线程上其他 Channel 的消息处理也堆积。这就是简历中提到的"伪同步 ACK"瓶颈。改进方案：1) 异步 Feign 调用；2) 先写 Redis 缓存 ACK 状态，再异步批量回写 MySQL；3) 用 MQ 异步投递 ACK 请求）

**为什么同时支持两种 ACK 方式？什么场景用 seq，什么场景用 messageId？**
（答：seq 用于批量 ACK——"我收到了这个 session 到 seq=100 的所有消息"，一次 ACK 顶多条，效率高。messageId 用于单条精确 ACK——某些场景客户端可能只知道 messageId 不知道 seq。服务端 ackByMessageId 需要先回查消息获取 seq，多一次 DB 查询，效率较低）

**如果 ACK 丢了怎么办？消息会重复推送吗？**
（答：ACK 丢失 → 服务端不知道客户端已收到 → lastAckSeq 没更新 → 下次重连补拉时会重新拉到这些消息 → 重复推送。客户端通过 messageId 或 seq 去重，相同消息不重复渲染。这就是"至少一次投递 + 消费端去重"语义）

十四、压测相关

14.1 你简历中的压测数据

> RTC 1000 长连接稳定在线，高频心跳场景无异常断连
> 节点重启后约 8s 恢复至 100% 活跃连接

**1000 长连接是用什么工具建立的？**
（答：自写 Node.js 脚本 ws_load_test.js，基于 ws 库。不用 k6/wrk 是因为它们是 HTTP 压测工具，不直接支持 WebSocket 长连接的心跳保活、断线重连、ACK 应答等行为模拟。脚本支持参数化配置：连接数、心跳频率、退避策略、ramp-up 阶梯建连等）

**怎么验证"无异常断连"？监控指标是什么？**
（答：脚本每 5 秒输出一行 STAT，包括 active/attempted/opened/closed/errors/unexpected 等累计指标。稳定在线 = active 持续等于目标连接数 && closed=0 && errors=0 && unexpected=0。服务端同时用 jstat -gcutil 观察 GC 行为，确保没有 Full GC 导致的停顿）

**"高频心跳场景"的频率是多少？**
（答：默认客户端心跳 30s 一次（heartbeatMs=30000），高频测试时改为 10s 一次（--heartbeat-ms=10000）。1000 连接 × 10s 心跳 = 100 次/秒心跳消息。压测期间只有正常 Young GC，没有新增 Full GC，Old Gen 稳定在约 15%）

**8 秒恢复是怎么测的？**
（答：从 kill -9 进程到脚本输出 active=1000/1000 的时间差。优化前固定延迟约 11 秒恢复，优化后指数退避 + 抖动约 8 秒恢复）

**为什么只测了 1000 连接？单节点理论上限是多少？**
（答：受限于 2C8G 机器配置，且 JVM 堆只给了 -Xms256m -Xmx512m（同一台机器还跑着 Gateway 和 Messaging）。理论上 Netty 单机百万连接，但内存、文件描述符、CPU 都是瓶颈。当前 512MB 堆、Old Gen 15%（约 76MB）的情况下，1000 连接的这轮压测目的不是追上限，而是验证接入层稳定性和恢复能力）

**压测时监控了哪些指标？**
（答：1) 客户端侧：active/attempted/errors/opened/closed/unexpected 连接指标；2) 服务端 JVM：jstat -gcutil 观察 YGC/FGC 频率和 Old Gen 占比；3) 系统级：CPU、内存、文件描述符数（ulimit -n）、TCP 连接状态分布（netstat/ss 看 TIME_WAIT 数量）、Redis 连接数（INFO clients））

RTC 服务启动命令：
```bash
nohup java -Xms256m -Xmx512m \
  -Dspring.cloud.nacos.discovery.ip=172.31.130.141 \
  -jar /opt/infinitechat/RealTimeCommunication-0.0.1-SNAPSHOT.jar \
  > /opt/infinitechat/logs/rtc.log 2>&1 &
```

**512MB 堆能撑多少连接？**
（答：每个 Netty Channel + Pipeline + Attribute 约占 2~5KB 堆内存。1000 连接约 2~5MB。Old Gen 15% = 76MB，大部分是 Spring 容器 + 类加载 + Redis/MQ 客户端。连接本身占比很小。粗估 512MB 堆可以撑 1 万+ 连接，但需要压测验证——ByteBuf、TaskQueue 等间接内存消耗也要考虑）

**为什么不给更大的堆？**
（答：2C8G 机器上同时跑 Gateway + Messaging + RTC 三个 JVM，需要分配内存。512MB 是在当前部署密度下给 RTC 的配额。8GB 物理内存 - OS 占用 ~1GB - 三个 JVM 各 512MB~1GB ≈ 刚好用完）

**Netty 的 ByteBuf 默认走 Direct 还是 Heap？DirectMemory 默认上限是多少？**
（答：Netty 4.x 默认用 PooledByteBufAllocator + Direct Memory。DirectMemory 默认上限 = -Xmx 的值 = 512MB。Direct 不受 GC 管理，通过 Cleaner 释放。需要关注 -XX:MaxDirectMemorySize 参数，如果 Direct 占用过多会报 OutOfDirectMemoryError）

十五、生产环境优化与容灾（加分题）

**如果要支持 10 万连接，架构怎么改？**
（答：水平扩容多个 Netty 节点 + 网关层基于 Redis ZSet 的最小连接数负载均衡 + Redis Cluster 支撑路由表。每个节点 1~2 万连接，5~10 个节点。JVM 堆调大到 2~4GB，Worker 线程 8~16 个）

**如何实现连接的平滑迁移？不中断用户体验的情况下把连接从节点 A 迁到节点 B？**
（答：1) 节点 A 发送"请重连"指令给客户端；2) 节点 A 从 ZSet 摘除自己避免接收新连接；3) 客户端收到指令后主动断开重连，Gateway 把新连接分到节点 B；4) 节点 A 等所有连接迁走后关闭。这就是灰度下线方案）

**Netty 节点和 Spring Boot HTTP 服务跑在同一个 JVM 里，有什么隐患？**
（答：GC 停顿会同时影响 WebSocket 推送和 HTTP 请求处理。HTTP 如果有慢请求或 OOM，长连接也会受影响。512MB 堆下 GC 频率更高。生产建议拆分成两个进程，或至少加大堆到 2GB+）

**你的项目没有用 SSL/TLS（wss），生产环境怎么加？**
（答：两种方案：1) 在 Netty Pipeline 最前面加 SslHandler（基于 OpenSSL 或 JDK SSL），性能好但证书管理在应用层；2) 在前面加一层 Nginx 做 TLS 终结，Netty 只处理明文。方案 2 更常见——Nginx 做证书管理和 TLS 卸载，Netty 专注业务。性能差异：Nginx TLS 终结会多一次代理开销，但可以利用 Nginx 的 TLS 加速能力）
