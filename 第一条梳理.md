2.1 Pipeline 中每个 Handler 的职责
你的 Pipeline 顺序（NettyServer.java:78-85）：

顺序	Handler	所在线程组	职责
1	IdleStateHandler(300, 0, 0)	Worker	读空闲检测，5分钟无数据触发事件

可能被问到的问题：

为什么 IdleStateHandler 放在最前面？ 如果放在 HttpServerCodec 之后呢？（答：放在最前面可以在字节级别检测空闲，不依赖编解码是否成功。放在编解码后面也可以，但如果编解码异常阻塞了，idle 检测就失效了）

为什么 IM_CHAT 和 IM_NOTIFY 要分两个 Topic？ 能不能合一个？（答：隔离不同业务的消费速率和重试策略。聊天消息是核心链路不能丢，通知类容忍度更高）
selectorExpression = "${im.node.id:NODE_1}"（NettyPushMessageListener.java:27）意味着每个节点只消费 tag=自己nodeId 的消息。那如果某个节点宕机了，它 tag 的消息谁消费？（答：没人消费，堆积在 Broker 直到超时。消息发送端已经做了路由查询，如果用户不在线会走 OFFLINE tag。但如果消息发送时用户在线，发完后节点马上宕机呢？→ 消息丢失。需要消费端 ACK + 重试机制）
ConsumeMode 为什么用 CONCURRENTLY 而不是 ORDERLY？ 消息顺序怎么保证？（答：聊天消息的顺序在发送端通过 seq 保证，消费端只负责推送，不需要严格顺序。如果推送失败回流离线后重拉时靠 seq 排序）
离线回流的消息怎么最终送达用户？ 用户重连后怎么触发补拉？（答：用户上线后主动拉取离线消息，基于 sessionId + lastAckSeq 做增量拉取）

| reconnectDelayMs | 1000ms | 初始重连延迟 |
| reconnectMaxDelayMs | 10000ms | 最大重连延迟（封顶） |
| reconnectBackoffFactor | 2 | 指数底数 |
| reconnectJitterMs | 1000ms | 随机抖动上限 |
| maxReconnectAttempts | 0（无限） | 不限重试次数 |

退避公式（ws_load_test.js:211-228）：
```
cappedDelay = min(baseDelay × 2^(attempt-1), maxDelay)
finalDelay  = uniform_random(minDelay, cappedDelay + jitter)
```

具体退避轨迹：
| 重试次数 | cappedDelay | 实际范围（加 jitter） |
|---------|-------------|---------------------|
| 第1次 | 1s | 1s ~ 2s |
| 第2次 | 2s | 1s ~ 3s |
| 第3次 | 4s | 1s ~ 5s |
| 第4次 | 8s | 1s ~ 9s |
| 第5次+ | 10s（封顶） | 1s ~ 10s |

2000 次降到 126 次是怎么来的？

优化前（固定 1s 延迟，无退避无抖动）实际压测日志：
```
04:03:50 active=0/1000 attempted=1000 errors=0     ← 稳态 1000 连接，执行 kill 停机
04:03:53 active=0/1000 attempted=2000 errors=998    ← 第一波重连全失败（服务还没起来）
04:03:55 active=0/1000 attempted=2998 errors=1910   ← 第二波继续失败
04:03:57 active=0/1000 attempted=3000 errors=2000   ← 3 轮固定重试打完，累计 2000 失败
04:03:59 active=291/1000 attempted=4000 errors=2000 ← 服务恢复，291 先连上
04:04:01 active=1000/1000 attempted=4000 errors=2000 ← 全部恢复，kill → 满连约 11 秒
```

优化后（指数退避 + 随机抖动）：attempted=1126, errors=126, 约 8s 恢复满连。
核心收益不是恢复速度快了 3 秒，而是无效重试从 2000 降到 126——降低了恢复窗口内对服务端的冲击。

随机抖动（jitter）的目的是什么？ （答：防止多个客户端的退避时间完全对齐形成"周期性洪峰"。纯指数退避时 1000 个客户端的第 N 次重试仍然会在同一时刻到来，加 jitter 后重连请求被打散在时间轴上）

十三、ACK 机制
13.1 你的 ACK 方案（MessageInboundHandler.java:142-164）

客户端收到消息后发 ACK:
  方式1: { sessionId, seq } → Feign 调用 MessageService.ackBySeq()
  方式2: { messageId/msgUuid } → Feign 调用 MessageService.ackByMessageId()
可能被问到的问题：

ACK 处理走 Feign（MessageAckClient.java），这是同步 HTTP 调用，在 Business 线程里会阻塞吗？ （答：会阻塞 Business 线程直到 Feign 返回。如果 MessageService 响应慢，Business 线程池会耗尽。这是你简历中提到的"伪同步 ACK"瓶颈？应该改为异步，或者先写 Redis 再异步批量回写 MySQL）
为什么同时支持两种 ACK 方式？ 什么场景用 seq，什么场景用 messageId？（答：seq 用于批量 ACK——"我收到了这个 session 到 seq=100 的所有消息"，一次 ACK 顶多条。messageId 用于单条精确 ACK）
如果 ACK 丢了怎么办？ 消息会重复推送吗？（答：会。这就是"至少一次投递 + 消费端去重"语义。客户端通过 seq 或 messageId 去重）
