一、红包全链路总览（面试官第一问：画出发红包和抢红包的完整链路）

```
========== 发红包 ==========

客户端
  ↓ POST /api/message/redpacket/send (经 Gateway 鉴权)
RedPacketController.sendRedPacket()
  ↓ request.setSendUserId(userId) — X-User-Id 由网关注入，防伪造
RedPacketServiceImpl.sendRedPacket()  ← 整个方法在 @Transactional 内
  ├── 1. validateSendRedPacketRequest() — 金额/数量/类型校验
  ├── 2. deductUserBalanceSafe(senderId, totalAmount)
  │    → UPDATE user_balance SET balance = balance - ? WHERE user_id = ? AND balance >= ?
  │    → 返回 0 → 余额不足，抛异常，整个事务回滚
  ├── 3. createRedPacket() → INSERT red_packet（雪花ID, 状态=UNCLAIMED）
  ├── 4. createBalanceLog() → INSERT balance_log（type=SEND, amount=负数）
  ├── 5. sendRedPacketMessage() → 复用 MessageService.sendMessage()
  │    → type=5(红包), clientMsgId="rp_"+redPacketId, body={content:红包ID, 祝福语}
  ├── 6. preCalculateAndStoreAmounts() → Redis 预计算金额
  │    ├── red_packet:count:{id} = totalCount, TTL=24h
  │    └── red_packet:amounts:{id} = [金额1, 金额2, ...], TTL=24h
  └── 7. sendExpireDelayMessage() → RocketMQ 延迟消息，24h 后投递到 RED_PACKET_EXPIRE

========== 抢红包 ==========

客户端
  ↓ POST /api/message/redpacket/receive
RedPacketReceiveServiceImpl.receiveRedPacket()
  ├── 1. 极速拦截：GET red_packet:count:{id}, count<=0 直接返回"已抢完"
  ├── 2. 生成 corrId(UUID), 创建 CompletableFuture 挂在 FUTURE_CACHE
  ├── 3. redPacketTxRocketMQTemplate.sendMessageInTransaction(
  │      "IM_RED_PACKET_RECEIVE", mqMsg, [redPacketId, userId, corrId])
  │         ↓ 触发
  │    RedPacketTxListener.executeLocalTransaction()
  │      ├── 执行四合一 Lua 脚本 (UNIFIED_GRAB_LUA)
  │      │    → 返回 "-1" → 已领取过 → future.complete(-1) → ROLLBACK
  │      │    → 返回 "0"  → 已抢完   → future.complete(0)  → ROLLBACK
  │      │    → 返回 金额  → 抢成功   → future.complete(金额) → COMMIT
  │      └── COMMIT 的消息投递到 IM_RED_PACKET_RECEIVE topic
  ├── 4. future.get(3, SECONDS) — HTTP 线程阻塞等待 Lua 结果
  └── 5. 返回 ReceiveRedPacketResponse{amount, status}

         ↓ MQ 异步（COMMIT 后）
RedPacketReceiveListener.onMessage()
  ├── 从 Redis Pending Hash 读取金额
  ├── doSaveReceiveRecord()  ← @Transactional
  │    ├── 幂等检查: countByRedPacketAndUser > 0 → 跳过
  │    ├── 行锁扣减: deductRedPacketRowLock(id, amount)
  │    ├── 插入领取记录: INSERT red_packet_receive
  │    ├── 加用户余额: addBalanceAtomically(userId, amount)
  │    └── 写资金流水: INSERT balance_log（type=RECEIVE）
  └── 事务提交后: 清理 Redis Pending Hash

========== 红包过期 ==========

方式1: RocketMQ 延迟消息 → RED_PACKET_EXPIRE topic → RedPacketExpireListener
方式2: 定时任务 RedPacketExpireTask, 每小时扫描兜底

handleExpiredRedPacket()
  ├── CAS 更新: status UNCLAIMED→EXPIRED（幂等）
  ├── 退还剩余金额: addBalanceAtomically(senderId, remainingAmount)
  └── 写退款流水: INSERT balance_log（type=REFUND）
```

可能被问到的问题：

- **为什么发红包用 @Transactional，而抢红包不在一个事务里？** （答：发红包的"扣余额 + 创建红包 + 记流水"必须原子性——扣了钱红包没创建是不行的。抢红包的核心逻辑在 Redis Lua 脚本中完成（毫秒级），MySQL 落库是异步的，用 MQ 事务消息保证最终一致性）
- **整个抢红包链路的延迟在哪里？** （答：客户端视角：HTTP 请求 = 极速拦截(Redis GET) + 发事务消息 + Lua 执行 + CompletableFuture 等待。Lua 执行在微秒级，事务消息发送约 1~3ms。总延迟约 5~10ms。MySQL 落库是异步的，不在请求路径上）

二、四合一 Lua 原子抢红包脚本（核心亮点）

2.1 脚本全文（RedPacketConstants.UNIFIED_GRAB_LUA）

```lua
-- KEYS[1] = red_packet:received:{redPacketId}   已领取用户 Set
-- KEYS[2] = red_packet:count:{redPacketId}       剩余个数
-- KEYS[3] = red_packet:amounts:{redPacketId}     预计算金额 List
-- KEYS[4] = red_packet:pending:{redPacketId}     待落库暂存 Hash
-- ARGV[1] = userId

-- 第1步：防重领——检查用户是否已领过
if redis.call('sismember', KEYS[1], ARGV[1]) == 1 then return '-1' end

-- 第2步：检查库存
local count = redis.call('get', KEYS[2])
if count == false or tonumber(count) <= 0 then return '0' end

-- 第3步：弹出预计算金额（LPOP 原子弹出，先到先得）
local amount = redis.call('lpop', KEYS[3])
if amount == false then return '0' end

-- 第4步：四步原子操作
redis.call('decr', KEYS[2])                    -- 库存 -1
redis.call('sadd', KEYS[1], ARGV[1])           -- 记录已领取
redis.call('hset', KEYS[4], ARGV[1], amount)   -- 写入暂存 Hash
return amount                                    -- 返回抢到的金额
```

2.2 四个 Redis Key 的生命周期

| Key | 类型 | 创建时机 | 销毁时机 | TTL |
|-----|------|---------|---------|-----|
| `red_packet:count:{id}` | String | 发红包时 preCalculateAndStoreAmounts | 自然过期 | 24h |
| `red_packet:amounts:{id}` | List | 发红包时 RPUSH 预计算金额 | LPOP 到空 / 自然过期 | 24h |
| `red_packet:received:{id}` | Set | 第一个人抢到时 SADD | 自然过期 | 无（依赖 amounts 的 TTL） |
| `red_packet:pending:{id}` | Hash | Lua 脚本 HSET | RedPacketReceiveListener 落库后 HDEL | 无 |

可能被问到的问题：

- **为什么要把四步操作合在一个 Lua 脚本里？** （答：Redis 单线程执行 Lua 脚本是原子的。如果拆成多个命令：1) SISMEMBER 检查没领过，2) DECR 库存，3) LPOP 金额——两个用户并发时可能同时通过步骤 1 的检查，都执行步骤 2 和 3，导致超发。Lua 脚本保证 check-then-act 的原子性）
- **LPOP 弹出金额后，如果后续 MySQL 落库失败怎么办？金额不就丢了？** （答：不会丢。Lua 脚本在 LPOP 的同时把金额写入 Pending Hash（HSET）。即使 MySQL 落库失败，Pending Hash 里还保留着金额。Broker 回查时检查 Pending Hash 存在 → COMMIT → 重新消费 → 再次尝试落库。Pending Hash 是"临时保险箱"）
- **`red_packet:received:{id}` 这个 Set 没有设 TTL，会不会内存泄漏？** （答：会。红包领完后 Set 一直在 Redis 里。需要在红包过期处理时主动 DEL 清理，或者在 preCalculateAndStoreAmounts 时给 received Set 也设 24h TTL。当前代码确实没有清理）
- **如果 Redis 宕机了，所有红包都抢不了？** （答：是的。抢红包完全依赖 Redis。发红包时 MySQL 已经记录了红包信息（remaining_amount/remaining_count），Redis 宕机恢复后需要从 MySQL 重建 Redis 数据。但当前代码没有这个重建机制）
- **为什么用 LPOP 而不是随机弹出？** （答：金额在发红包时已经预计算好并 shuffle 了（拼手气红包），存入 List 的顺序就是随机的。LPOP 是按顺序弹出——第一个抢的拿第一个金额，第二个抢的拿第二个。因为 List 已经 shuffle 过，等价于随机分配）

三、RocketMQ 事务消息机制详解

3.1 为什么抢红包需要事务消息？

核心问题：**Redis Lua 脚本执行成功（预扣库存）后，必须保证 MySQL 落库也成功**

- 如果不用事务消息：Lua 成功 → 发普通 MQ → MQ 丢了 → Redis 已扣但 MySQL 没扣 → 数据不一致
- 用事务消息的流程：
  ```
  1. 发送 Half Message（MQ 暂不投递）
  2. 执行本地事务（Lua 脚本）
  3a. Lua 成功 → COMMIT → Half Message 投递给消费者 → 异步落库
  3b. Lua 失败 → ROLLBACK → Half Message 丢弃
  4. 如果 3 没有返回 → Broker 回查 → 检查 Pending Hash → 决定 COMMIT/ROLLBACK
  ```

3.2 事务消息执行流程（RedPacketTxListener.java）

```java
// executeLocalTransaction: 在发送 Half Message 之后立即执行
public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
    Object[] args = (Object[]) arg;
    Long redPacketId = (Long) args[0];
    Long userId = (Long) args[1];
    String corrId = (String) args[2];

    // 执行四合一 Lua 脚本
    String result = redisTemplate.execute(GRAB_SCRIPT, keys, userId.toString());

    // 通过 CompletableFuture 唤醒挂起的 HTTP 线程
    CompletableFuture<BigDecimal> future = FUTURE_CACHE.get(corrId);

    if ("0".equals(result)) {
        future.complete(BigDecimal.ZERO);    // 已抢完
        return ROLLBACK;
    }
    if ("-1".equals(result)) {
        future.complete(new BigDecimal("-1")); // 已领取过
        return ROLLBACK;
    }

    BigDecimal amount = new BigDecimal(result);
    future.complete(amount);                   // 抢到了
    return COMMIT;                             // 消息投递给消费者
}
```

3.3 Broker 回查（RedPacketTxListener.java:94-113）

```java
// checkLocalTransaction: Broker 在 15s/30s/60s... 后回查未确认的事务
public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
    String pendingKey = "red_packet:pending:" + redPacketIdStr;
    Object val = redisTemplate.opsForHash().get(pendingKey, userIdStr);
    if (val != null) {
        return COMMIT;    // Pending Hash 有记录 → Lua 执行成功 → 补发 COMMIT
    }
    return ROLLBACK;      // Pending Hash 无记录 → Lua 未执行或已回滚 → 补发 ROLLBACK
}
```

可能被问到的问题：

- **CompletableFuture + FUTURE_CACHE 是干什么的？** （答：HTTP 请求线程（Tomcat 线程）需要知道 Lua 脚本的执行结果（抢到了多少钱）。但 Lua 脚本是在事务消息的 executeLocalTransaction 回调中执行的（不同线程）。用 CompletableFuture 做线程间通信：HTTP 线程 future.get(3s) 阻塞等待，Lua 线程 future.complete(amount) 唤醒）
- **corrId 是什么？为什么用 UUID？** （答：correlationId，关联 ID。每次抢红包请求生成一个唯一的 corrId，作为 FUTURE_CACHE 的 key。HTTP 线程和事务线程通过同一个 corrId 找到同一个 CompletableFuture。UUID 保证全局唯一）
- **future.get(3, SECONDS) 超时后怎么办？** （答：抛 TimeoutException，返回"系统繁忙，请重试"。但 Lua 脚本可能已经执行成功了——金额已弹出、库存已扣减。此时消息会 COMMIT，消费者异步落库。用户虽然看到"请重试"，但实际上已经抢到了。下次查看红包详情时能看到领取记录。不会超发）
- **FUTURE_CACHE 是 ConcurrentHashMap，有没有内存泄漏风险？** （答：有。finally 块中 FUTURE_CACHE.remove(corrId) 保证正常/异常都清理。但如果 executeLocalTransaction 很久没被调用（比如 MQ 连接断了），future 会超时被 get 那边抛异常，finally 也会清理。所以理论上不会泄漏）
- **事务消息和普通消息的区别？** （答：普通消息发送后立即投递给消费者。事务消息分两阶段：1) Half Message 对消费者不可见；2) 执行本地事务后 COMMIT/ROLLBACK 决定是否投递。Broker 有回查机制保证最终一致性）

四、发红包链路深挖

4.1 余额扣减（UserBalanceMapper.java:21-22）

```sql
UPDATE user_balance
SET balance = balance - #{amount}, updated_at = NOW()
WHERE user_id = #{userId} AND balance >= #{amount}
```

可能被问到的问题：

- **`balance >= #{amount}` 这个 WHERE 条件是干什么的？** （答：防止余额变负。如果用户余额 100 元，两个请求同时扣 80 元：第一个 100 >= 80 成功，余额变 20；第二个 20 >= 80 失败，返回 updateCount=0。这就是 **CAS（Compare-And-Swap）** 模式在 SQL 中的体现）
- **这个 UPDATE 是行锁吗？** （答：是。InnoDB 在 UPDATE 时对匹配到的行加排他锁（X Lock）。同一用户的多个扣减请求会串行执行，不会并发扣减导致超扣）
- **@Transactional 中先扣余额再创建红包，如果创建红包失败了？** （答：事务回滚，余额扣减也回滚。这就是 @Transactional 的作用——保证"扣余额 + 创建红包 + 记流水"的原子性）

4.2 金额预计算

**普通红包（generateNormalAmounts）**：平均分配
```
10 元 / 3 个 = 3.33, 3.33, 3.34（最后一个拿剩余）
```

**拼手气红包（generateRandomAmounts）**：二倍均值法
```
算法：每次随机金额 ∈ [minAmount, 剩余平均值×2]
示例：10 元 5 个
  → 平均值 = 2.00, max = 4.00, 随机出 2.73
  → 剩余 7.27 / 4 = 1.82, max = 3.63, 随机出 1.55
  → ...
  → 最后一个拿剩余
  → 打乱顺序（Collections.shuffle）
```

可能被问到的问题：

- **二倍均值法的数学证明？为什么能保证公平？** （答：每个人抢到的期望金额 = 剩余金额 / 剩余人数 = 平均值。随机范围 [min, 2×平均值]，期望值 = 平均值。后面的人不会因为前面的人抢了大金额而吃亏——因为他们的范围也缩小了，但期望值不变）
- **为什么最后要 shuffle？** （答：二倍均值法生成的金额序列有规律——前面的金额波动大，后面的金额趋向平均。shuffle 后存入 Redis List，LPOP 取出的顺序变成随机的，谁先抢到不影响公平性）
- **minAmount = 0.01 元，如果 10 元发 1000 个包怎么办？** （答：validateSendRedPacketRequest 校验：平均金额 = totalAmount / totalCount。如果 10/1000 = 0.01 刚好等于 minAmount，可以发。但每个红包都是 0.01 元，没有拼手气的空间。如果 totalAmount 更小就会被拦截）

@Transactional(rollbackFor = Exception.class)
public void doSaveReceiveRecord(Long redPacketId, Long userId, BigDecimal amount) {
    // 1. 幂等检查
    int exists = redPacketReceiveMapper.countByRedPacketAndUser(redPacketId, userId);
    if (exists > 0) return;  // 已处理过

    // 2. 行锁扣减红包库存
    int rows = redPacketMapper.deductRedPacketRowLock(redPacketId, amount);
    // SQL: UPDATE red_packet SET remaining_amount -= ?, remaining_count -= 1
    //      WHERE red_packet_id = ? AND remaining_count > 0 AND remaining_amount >= ?

    // 3. 插入领取记录（唯一索引兜底）
    RedPacketReceive receive = new RedPacketReceive();
    receive.setRedPacketId(redPacketId);
    receive.setReceiverId(userId);
    receive.setAmount(amount);
    redPacketReceiveMapper.insert(receive);

    // 4. 加用户余额
    userBalanceMapper.addBalanceAtomically(userId, amount);
    // SQL: UPDATE user_balance SET balance = balance + ?

    // 5. 写资金流水
    balanceLogMapper.insert(balanceLog);  // type=RECEIVE_RED_PACKET
}
```

可能被问到的问题：

- **Pending Hash 在事务边界外清理，为什么？** （答：如果在事务内清理，事务回滚了但 Pending Hash 已经删了 → 金额丢失。在事务提交后清理，保证"DB 落库成功 → 再清 Pending Hash"。如果清理失败，Pending Hash 残留 → Broker 回查时认为已处理 → COMMIT → 再次消费 → 幂等检查发现已存在 → 跳过 → 最终清理）
- **deductRedPacketRowLock 的 WHERE 条件 remaining_count > 0 AND remaining_amount >= #{amount}，和 Redis 的库存检查是双重保障吗？** （答：是。Redis Lua 是第一道防线（微秒级），MySQL 行锁是第二道防线（毫秒级）。即使 Redis 数据和 MySQL 数据不一致（比如 Redis 重建后），MySQL 的 WHERE 条件确保不超发）
- **DuplicateKeyException catch 是什么？** （答：red_packet_receive 表有 (red_packet_id, receiver_id) 唯一索引。如果 MQ 重复消费导致重复 INSERT，唯一索引会报冲突。catch 后跳过，保证幂等。这是第三层防重——Lua Set 防重 → 应用层 count 防重 → 唯一索引防重）

六、红包过期与退款

6.1 双重保障机制

```
机制1: RocketMQ 延迟消息 (主路径，精确)
  发红包时 → syncSendDeliverTimeMills("RED_PACKET_EXPIRE", msg, now + 24h)
  24h 后投递 → RedPacketExpireListener → handleExpiredRedPacket()

机制2: 定时任务兜底扫描 (备路径，兜底)
  @Scheduled(cron = "0 0 */1 * * ?")  — 每小时

  → handleExpiredRedPacket()
```

6.2 handleExpiredRedPacket（RedPacketServiceImpl.java:362-408）

```java
@Transactional
public void handleExpiredRedPacket(Long redPacketId) {
    // CAS 更新状态：UNCLAIMED → EXPIRED（幂等——已领完/已过期的红包 rows=0）
    int rows = redPacketMapper.casUpdateStatus(redPacketId, UNCLAIMED, EXPIRED);
    if (rows == 0) return;  // 幂等跳过

    RedPacket redPacket = getById(redPacketId);
    BigDecimal remainingAmount = redPacket.getRemainingAmount();
    if (remainingAmount > 0) {
        // 退还剩余金额给发送者
        userBalanceMapper.addBalanceAtomically(senderId, remainingAmount);
        createBalanceLog(senderId, remainingAmount, REFUND_RED_PACKET, redPacketId);
    }
}
```

casUpdateStatus SQL：
```sql
UPDATE red_packet SET status = #{targetStatus}
WHERE red_packet_id = #{id} AND status = #{expectedStatus}
```

可能被问到的问题：

- **CAS 更新是什么？为什么不直接 UPDATE SET status=EXPIRED？** （答：CAS = Compare-And-Swap。WHERE status = UNCLAIMED 确保只有"未领完"状态的红包才能被标记为过期。如果红包已经被全部领取（status=CLAIMED），CAS 返回 0，跳过退款。防止重复退款）
- **延迟消息和定时任务都可能触发 handleExpiredRedPacket，会不会重复退款？** （答：不会。CAS 是幂等的——第一次成功，第二次 rows=0 跳过。无论触发几次，退款只执行一次）
- **syncSendDeliverTimeMills 是 RocketMQ 5.x 的精确定时消息？** （答：是。与旧版 delayLevel（18 个固定档位）不同，deliverTimeMills 支持任意毫秒级精度的定时投递。需要 Broker 开启 timerWheelEnable=true）
- **如果延迟消息丢了呢？** （答：定时任务每小时兜底扫描。最坏情况下红包在 25 小时后才被标记过期（24h + 1h 扫描间隔）。对用户来说可接受）

七、资金安全保障（面试重点追问）

7.1 防超发（最核心的安全问题）

三层防线：
| 层级 | 机制 | 代码位置 |
|------|------|---------|
| 第1层 | Redis Lua 原子脚本：DECR count + LPOP amount | RedPacketTxListener |
| 第2层 | MySQL 行锁：remaining_count > 0 AND remaining_amount >= amount | RedPacketMapper.deductRedPacketRowLock |
| 第3层 | 金额预计算：所有金额之和 = totalAmount，不存在多弹金额 | RedPacketServiceImpl.preCalculateAndStoreAmounts |

7.2 防重复领取

三层防线：
| 层级 | 机制 | 代码位置 |
|------|------|---------|
| 第1层 | Redis Set SISMEMBER 检查 | Lua 脚本第 1 步 |
| 第2层 | 应用层 COUNT 查重 | doSaveReceiveRecord 第 1 步 |
| 第3层 | 数据库唯一索引 (red_packet_id, receiver_id) | INSERT catch DuplicateKeyException |

7.3 资金流水完整性

| 操作 | balance_log.type | amount 符号 | related_id |
|------|-----------------|-------------|------------|
| 发红包 | 1 (SEND) | 负数 | 红包 ID |
| 领红包 | 2 (RECEIVE) | 正数 | 红包 ID |
| 退红包 | 3 (REFUND) | 正数 | 红包 ID |

可能被问到的问题：

- **如何验证"零超发"？** （答：压测后对账：SUM(red_packet_receive.amount) WHERE red_packet_id=? 是否等于 totalAmount - remainingAmount。如果所有红包的这两个值都相等，就是零超发）
- **如何验证"零重复领取"？** （答：SELECT red_packet_id, receiver_id, COUNT(*) FROM red_packet_receive GROUP BY 1,2 HAVING COUNT(*) > 1。如果结果为空，就是零重复）
- **如果 Redis 和 MySQL 的数据不一致怎么办？** （答：以 MySQL 为准。Redis 是缓存/加速层，MySQL 是真相源。定期对账：比较 Redis count 和 MySQL remaining_count，如果不一致则告警并从 MySQL 重建 Redis）

九、RocketMQ 事务消息 vs 本地消息表

9.1 对比

| 维度 | 事务消息（你的抢红包方案） | 本地消息表（你的消息发送方案） |
|------|------------------------|--------------------------|
| 保证 | Broker 回查保证最终一致 | 定时任务重试保证最终一致 |
| 延迟 | 毫秒级（Half Message → COMMIT） | 秒级~分钟级（定时扫描间隔） |
| 复杂度 | 高（TxListener + 回查 + CompletableFuture） | 中（额外一张表 + 定时任务） |
| 适用 | "本地操作 + MQ 投递"需要原子性 | MQ 投递失败的简单兜底 |

9.2 为什么抢红包用事务消息而不是本地消息表？

（答：抢红包的"本地操作"是 Redis Lua 脚本，不是 MySQL 事务。本地消息表方案要求本地操作和消息写入在同一个 MySQL 事务中。而 Redis 操作不能纳入 MySQL 事务。RocketMQ 事务消息允许 executeLocalTransaction 执行任意本地操作（包括 Redis），再根据结果决定消息是否投递）

十、红包详情查询优化

10.1 GetRedPacketServiceImpl

```java
// 1. Redis 缓存查询，1小时 TTL
String cacheKey = "red_packet:detail:" + redPacketId + ":" + pageNum;

// 2. 缓存未命中 → MySQL
RedPacket redPacket = redPacketService.getById(redPacketId);
Page<RedPacketReceive> receivePage = redPacketReceiveMapper.selectByRedPacketId(page, redPacketId);

// 3. 批量查用户信息，消除 N+1
List<Long> userIds = records.stream().map(::getReceiverId).distinct().collect(toList());
List<User> users = userService.listByIds(userIds);  // 1条 SQL: WHERE id IN (...)
Map<Long, User> userMap = users.stream().collect(toMap(::getUserId, identity()));
// O(1) 查 user
```

可能被问到的问题：

- **N+1 问题是什么？你怎么解决的？** （答：N+1 = 查 1 次红包 + 每条领取记录查 1 次用户 = N+1 次 SQL。你的方案：1 次查红包 + 1 次查领取记录 + 1 次批量查所有相关用户 = 3 次 SQL。用 listByIds(IN 查询) + HashMap O(1) 查找消除 N+1）
- **缓存 key 包含 pageNum，翻页时每页独立缓存？** （答：是。每页的领取记录不同，需要独立缓存。但如果红包还在被领取中，缓存可能包含过时的 remainingCount/remainingAmount。1小时 TTL 后自动失效刷新）

十一、压测相关

11.1 简历数据

> 红包接口在 800 RPS 下 P99 51ms、零错误，一致性验证零超发、零重复领取

可能被问到的问题：

- **800 RPS 是发红包还是抢红包？** （答：抢红包。抢红包是高并发热点——群里一个红包几十人同时抢。发红包是低频操作）
- **怎么压测抢红包？** （答：1) 先发一批大红包（比如 100 个红包，每个 100 份）；2) wrk2 模拟大量用户并发抢；3) 压测后对账验证一致性）
- **P99 51ms 的延迟分布？** （答：抢红包的关键路径：Redis GET(count) + 事务消息发送(~2ms) + Lua 脚本执行(~0.1ms) + CompletableFuture 等待(~1ms) ≈ 5~10ms。P99 51ms 包含了少量慢请求——可能是 Redis 或 MQ 的尾延迟）
- **一致性验证怎么做的？** （答：压测结束后执行 SQL 对账：
  ```sql
  -- 验证零超发
  SELECT rp.red_packet_id,
         rp.total_amount,
         rp.remaining_amount,
         SUM(rpr.amount) as claimed_sum
  FROM red_packet rp
  LEFT JOIN red_packet_receive rpr ON rp.red_packet_id = rpr.red_packet_id
  GROUP BY rp.red_packet_id
  HAVING rp.total_amount != rp.remaining_amount + COALESCE(claimed_sum, 0);
  -- 结果为空 = 零超发

  -- 验证零重复领取
  SELECT red_packet_id, receiver_id, COUNT(*) c
  FROM red_packet_receive
  GROUP BY 1, 2
  HAVING c > 1;
  -- 结果为空 = 零重复
  ```）

十二、选型对比

12.1 抢红包方案对比

| 方案 | 你的实现 | 纯数据库行锁 | Redis + 普通 MQ | 分布式锁 |
|------|---------|-------------|----------------|---------|
| 并发处理 | Redis Lua (微秒级) | MySQL 行锁 (毫秒级) | Redis Lua + 异步 | Redisson 串行 |
| 一致性保证 | 事务消息 + Pending Hash | 事务保证 | 可能丢消息 | 强一致 |
| 性能 | 极高 | 中 | 高但有丢失风险 | 低 |
| 复杂度 | 高 | 低 | 中 | 中 |
| 适用 QPS | 万级 | 千级 | 万级 | 百级 |

12.2 金额分配算法对比

| 算法 | 二倍均值法（你的选择） | 线段切割法 | 随机+最后补齐 |
|------|---------------------|-----------|-------------|
| 公平性 | 高（期望相等） | 高 | 低（最后一个太大/太小） |
| 复杂度 | O(N) | O(N log N) | O(N) |
| 极端值 | 受控（最大 = 2×平均值） | 可能很大 | 可能很大 |
| 微信实现 | 据传微信用此算法 | — | — |

12.3 为什么不用 Redisson 分布式锁做抢红包？

（答：分布式锁是串行化的——同一时刻只有一个请求能处理。100 人同时抢红包需要串行 100 次。Redis Lua 脚本虽然也是单线程执行，但每次执行只需微秒级，100 次串行也只需 0.1ms。分布式锁的加锁/解锁开销远大于 Lua 脚本）

十三、RedPacketTxRocketMQTemplate

```java
@ExtRocketMQTemplateConfiguration(group = "${rocketmq.red-packet-tx-producer-group:im-red-packet-receive-tx-group}")
public class RedPacketTxRocketMQTemplate extends RocketMQTemplate { }
```

可能被问到的问题：

- **为什么需要单独的 RocketMQTemplate？** （答：RocketMQ 的事务消息需要独立的 Producer Group。默认的 RocketMQTemplate 用 im-message-producer-group 发普通消息。抢红包的事务消息用 im-red-packet-receive-tx-group。两个 Group 隔离，互不影响。@RocketMQTransactionListener 通过 rocketMQTemplateBeanName 绑定到对应的 Template）
- **如果不隔离会怎样？** （答：所有事务消息的 executeLocalTransaction 回调都会被同一个 Listener 处理。如果有多种事务消息（比如红包、转账），需要在 Listener 中按 Topic 分发，增加耦合。隔离后每种事务消息有自己的 Listener）

十五、生产环境优化与架构演进

面试官可能在以上基础上继续追问：

- **热点红包怎么处理？群里 10000 人同时抢一个红包** （答：Redis 单 key 在单线程下串行处理 EVAL，QPS 约 10 万+。10000 人并发不是问题。但如果有多个热点红包同时抢，可以考虑按 redPacketId hash 到不同 Redis 节点分散压力）
- **如果要支持"定向红包"（指定某些人可以抢）怎么改？** （答：在 Lua 脚本第 1 步之前增加一步：SISMEMBER red_packet:eligible:{id} userId，只有在"可抢名单"中的用户才能继续）
- **如果要支持"红包回退"（用户领了又退回）怎么做？** （答：需要新增 RETURN_RED_PACKET 流水类型。操作：扣用户余额 + 加红包 remainingAmount + 删 receive 记录 + 恢复 Redis（SADD 从 received 中移除，count 恢复）。Redis 恢复比较复杂，且金额已经 LPOP 了需要重新 RPUSH）
- **BigDecimal 的精度问题？浮点运算会不会出现 0.1 + 0.2 != 0.3 的问题？** （答：BigDecimal 不存在浮点精度问题。但 Lua 脚本中金额是以字符串形式传递的（"1.23"），不涉及浮点运算。MySQL DECIMAL(10,2) 也是精确的。唯一风险点：Redis ZSet score 用 double 存 seq 时有精度问题，但红包金额存在 List/Hash 中是字符串，无问题）
- **发红包和领红包在同一个 Messaging 服务中，如果 Messaging 宕机了？** （答：发红包不可用 + 抢红包不可用。Redis 中的预计算数据还在，重启后可继续抢。MySQL 中的红包记录还在，过期退款的定时任务也会继续工作。但如果是部署了多实例，宕一个不影响——Gateway 负载均衡到其他实例）
